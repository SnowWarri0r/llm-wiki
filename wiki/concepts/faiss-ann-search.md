# FAISS / 近似最近邻 · 用一点精度换十亿级的速度

## 一句话
精确最近邻要拿查询跟全部向量逐个比，十亿级太慢；FAISS 用"先分桶只翻几桶"(IVF)+"把向量压成几字节"(PQ)两招，换来毫秒级近似检索。

## 直觉 · 为什么不能精确比
[[siglip-semantic-dedup]] 去重、[[hierarchical-kmeans-curation]] 聚类，都要在十亿张图的向量里反复找"最近的邻居"。精确最近邻 = 查询向量跟**每一个**库向量算距离，取最小 → 十亿次距离、每次几百维乘加，慢到没法用。FAISS 的整个设计就是**用一点点精度（近似 ANN）换巨大的速度**，靠两个能叠用的招。

## 招 1 · IVF 倒排：别全找，先分桶
```
建索引: 先用 k-means 把全库聚成 ~√N 个"格子"(Voronoi cell)，各有一个中心
查询:   1) 查询先跟这 √N 个中心比，挑最近的 nprobe 个格子
        2) 只在这几个格子里的向量中找最近邻，其余格子整批跳过
```
像进图书馆找书：不挨本翻，先走到对的**书架**(格子)，只翻那几架。

## 招 2 · PQ 乘积量化：别存全量，压成几字节
```
把一个向量切成 m 段(子向量); 每段单独用 k-means 训一本 256 个中心的"码本"
每段 → 替换成"最近中心的编号"(0~255 = 1 字节)
于是整条向量 = m 个字节; 距离用预存的"段-距离查表"求和近似算
```
像把一长串颜色用"色卡编号"记：不存 RGB，存色卡上第几号，又小又能比。

两招叠起来（**IVF+PQ**）：先分桶跳掉绝大多数候选，再在要翻的桶里用压缩码算近似距离 → 十亿向量塞进几 GB、查询毫秒级。FAISS 还有 GPU kernel，再快 5~10×。

## 数字例子 · 跳过比 + 压缩比都算给你看
**IVF 省了多少次比较**（取 N=100 万）：
```
格子数 = √N = 1000，每格 ~1000 个向量
精确:  100 万次距离
IVF:   1000(先比中心) + nprobe×1000  ; nprobe=10 → 1000 + 10000 = 1.1 万次
       → 约 90× 更少（只翻 10/1000 = 1% 的库）
```
**PQ 压了多少**（768 维 float32）：
```
原始:  768 × 4 字节 = 3072 字节/向量
PQ:    切 8 段 → 每段 1 字节(256 码本) = 8 字节/向量
       → 3072 / 8 = 384× 压缩
十亿向量:  3.07 TB  →  8 GB   （从"放不进内存"变成"GPU 都装得下"）
```
✓ 自检：IVF 把"翻全库"变成"翻 1% 的库"(90× 提速)，PQ 把"3TB 存不下"变成"8GB 装得下"(384× 瘦身)——两件事正交，所以能乘起来用。代价是**近似**：可能漏掉极少数真正最近的邻居。旋钮就是 **nprobe**(翻几个桶)和 **PQ 段数/位数**：调大→更准但更慢/更占内存，调小→更快但召回降。本质是**召回率 ↔ 速度/内存**的权衡。

## 为什么 8 字节还够用 · "乘积"才是 PQ 的命门
8 字节才 256 种值？不——关键在 PQ 的 **product（乘积）**：8 段各自有 256 个码本中心，它们能**自由组合**。
```
存的零件:   8 段 × 256 中心 = 2048 个小向量（每个才 96 维）
能拼出的码: 256⁸ = 2⁶⁴ ≈ 1.8×10¹⁹ 种不同"整向量编码"
```
**存储是加法（8+256），表达力是乘法/指数（256⁸）**：用两千来个零件，拼出一千亿亿种货。要是直接对整向量做 k-means，想要 10¹⁹ 个中心是天方夜谭（存不下、也训不出）；一切段，这个组合爆炸就**白捡**了。所以"8 字节"不是"只能表示 256 类"，而是 8 个独立维度各挑一档 → 组合出天文数字级的精度。这就是 PQ 比普通向量量化高明的地方：**把高维空间拆成几个子空间各自量化、再让它们相乘**。

> 类比：车牌只有几十个字符，但"位数相乘"就能编出上亿张不重样——PQ 是同一招用在向量上。

## 跟 k-means 的关系
注意 k-means 在这里出现了**两次**、用途不同：IVF 用它把全库分成检索格子；PQ 用它在每个子段空间里训码本。[[hierarchical-kmeans-curation]] 里的 k-means 则是第三种用途（按语义分簇审查）——同一个算法，三处不同目的。
