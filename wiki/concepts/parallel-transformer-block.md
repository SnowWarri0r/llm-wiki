# 并行 Transformer 块 · attention 和 MLP 不接力,同时算再相加

## 一句话
普通 Transformer 块是"先注意力、再前馈"两路接力;并行块让两路**读同一份归一化输入、各算各的、最后一起加回主干**,省一次归一化、还能把两路的第一个大矩阵乘拼成一个 GEMM 一次算完——快 ~15%,表达力几乎不掉。GPT-J / PaLM / ViT-22B / [[flux-1]] 单流块都用它。

## 直觉 · 接力 vs 两个工人读同一张图
**串行块(标准)** 像流水线:A 工位(注意力)干完,产物传给 B 工位(MLP)接着干。B 必须等 A,因为 B 读的是"A 加工过的半成品"。

**并行块** 像两个工人**同时看同一张蓝图**(同一份归一化输入)各干各的,最后把两人的成果一起拼到主干上。没有谁等谁;而且两人"领料"那一步(各自的第一个矩阵乘)输入是同一份,可以**合并成一次领料**——这正是它在 GPU 上更快的根。

## 怎么做的 · 改了三点
串行块:
```
h   = x + Attention( LN1(x) )      # 第一棒
out = h + MLP( LN2(h) )            # 第二棒:MLP 吃的是含注意力结果的 h
```
并行块:
```
y   = LN(x)                        # ① 只归一化一次,两路共用 y
out = x + Attention(y) + MLP(y)    # ② MLP 吃 y(不是注意力输出)→ 无依赖,可同时
                                   # ③ 两路都加回同一个 x,加法可交换
```
三点改动:**一次 LN 顶两次** / **MLP 不再等 attention** / **两路结果加回同一主干**。前两点解锁并行,第三点保证顺序无所谓。

## 数字例子 · 把两路的矩阵乘拼成一个 GEMM
真正省时间的不是"逻辑上并行",而是**两路第一步输入都是 y → 权重拼成一个大矩阵一次算**。拿 [[flux-1]] 单流块的真实尺寸(hidden=3072、MLP 放大 4 倍):
```
两路第一步都从 y(3072 维)出发:
  注意力要 Q/K/V    → 3 × 3072  = 9216
  MLP 要升维(×4)   → 4 × 3072  = 12288
拼成一个 linear1:  3072 →(9216 + 12288)= 21504   ← 一次 GEMM 同时吐 QKV + MLP升维

算完按位置切开:前 9216 给注意力、后 12288 过 GELU 当 MLP 中间层
收尾再拼一次 linear2: [注意力输出 3072 | MLP 激活 12288] = 15360 → 3072
                                                ← 一次 GEMM 投影回主干并相加
```
✓ 自检:整块 = **2 个大矩阵乘 + 1 次注意力**;串行块是 **4 个分开的中等矩阵乘(QKV / 注意力投影 / MLP升维 / MLP降维)+ 2 次 LN**。GPU 吃"一个大 GEMM"远胜"几个小 GEMM"(kernel 启动少、算力利用率高),这一拼就是 ~15% 提速的来源。

## 代价 · 为什么不亏
唯一损失:并行块里 MLP 只看到原始 `y`,**对同层刚算出的注意力是瞎的**(串行块里它能看到)。但摞几十层后,**下一层的注意力照样吃到上一层 attention+MLP 的和**,丢的只是"同层内"那点先后,几乎不影响整体表达力——GPT-J(6B)、PaLM、ViT-22B 都验证过划算。所以 [[flux-1]] 只在最省的后段 38 个单流块用它,前段 19 个要对齐文图的双流块仍走标准串行(那里表达力更重要)。
