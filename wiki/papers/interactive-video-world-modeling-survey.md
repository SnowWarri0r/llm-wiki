# 交互式视频世界模型综述 · 不是“生成一段视频”，而是每次操作后都要继续维持同一个世界

文生视频像给导演一份剧本：条件在开拍前一次给完，生成结束后不能临时改。交互式视频世界模型更像正在运行的游戏：用户每隔一小段时间给动作，模型必须立刻生成下一段结果，而且桌子、人物、道路和已经发生的事不能凭空改变。

这篇综述覆盖游戏、开放世界、机器人和自动驾驶。若照论文顺序读，很容易被大量模型名淹没。更清楚的读法是先抓住三个互相拉扯的问题：

1. **动作听不听**：新输入能否真正改变下一帧；
2. **世界记不记得**：走远再回来，物体和空间是否还在；
3. **反馈够不够快**：模型能否在交互预算内给出结果。

## 一句话

**交互式视频世界模型是一个闭环生成器：它根据历史观察、历史动作和当前控制不断生成下一观察；现有方法主要在动作注入、长期记忆与少步/缓存加速之间做取舍。**

## 0. 先用“走进房间再转身”建立上下文

假设你操作一个第一人称角色：

```text
t=1：看见红沙发，按 W 前进
t=2：沙发变近，向右转
t=3：看见书架，继续前进
t=4：回头
```

一个合格的交互式世界模型至少要做到：

- `W` 真让镜头前进，而不是只生成一段“像在走”的视频；
- 右转后立刻看到房间右侧；
- 第 4 步回头时，红沙发仍在合理位置；
- 每一步都在可接受延迟内返回。

普通一次性视频生成只拿到“一个人在房间里行走”的开场条件，无法在 `t=2` 接收“现在右转”。这正是 **one-shot control** 与 **multi-shot interaction** 的区别。

## 1. 定义：下一帧由什么决定

论文把真实世界写成部分可观察马尔可夫决策过程 POMDP：

\[
(\mathcal S,\mathcal A,\mathcal O,\mathcal T,\mathcal R).
\]

不需要先学概率论，五个字母只是在列清单：

| 符号 | 人话 |
|---|---|
| `S` | 完整世界状态，例如全部物体位置；相机未必看得全 |
| `A` | 可执行动作，例如前进、转向、抓取 |
| `O` | 当前能观察到的图像、深度、声音等 |
| `T` | 世界怎样从当前状态变到下一状态 |
| `R` | 动作结果有多好，用于决策或强化学习 |

视频世界模型通常看不到完整 `s_t`，只能根据观察与动作生成下一观察：

\[
o_{t+1}\sim
p_\phi(o_{t+1}\mid H_t,a_t,c_t).
\]

- `o_{t+1}`：模型要生成的下一观察，常见是下一帧或下一小段视频；
- `p_φ`：参数为 `φ` 的生成分布；同样输入可能有多个合理未来，所以用分布而非等号；
- `a_t`：此刻用户或智能体给的动作；
- `c_t`：文字、音频、相机轨迹、布局等额外条件；
- `H_t`：截至当前的历史。

历史展开为：

\[
H_t=\{o_{1:t},a_{1:t-1},c_{1:t-1}\}.
\]

`1:t` 表示从第 1 步到第 `t` 步。注意当前动作 `a_t` 没放在 `H_t` 内，而是在条件栏单独出现：历史告诉模型“世界走到了哪里”，当前动作告诉模型“接下来往哪走”。

### 把例子代进去

在第 4 步回头时：

```text
H₄ = {
  o₁: 红沙发,
  o₂: 靠近沙发,
  o₃: 右侧书架,
  o₄: 书架近景,
  a₁: 前进,
  a₂: 右转,
  a₃: 前进
}
a₄ = 回头
```

模型生成 `o₅`。若历史记得好，`o₅` 应重新出现红沙发；若动作控制好，视角应按回头幅度变化；若系统够快，玩家才觉得这是操作而不是等待视频渲染。

## 2. 一套系统有六层，不是只有一个 DiT

把综述里的大量方法放进同一条链，会更容易比较：

```text
交互界面
键鼠 / 文字 / 拖拽 / 相机轨迹 / 机器人动作
        ↓
动作编码与注入
token 拼接 / scale-shift / cross-attention / 几何变换
        ↓
状态与记忆
历史帧 / latent memory / 3D 点图、Surfel、3DGS
        ↓
视频生成主干
VAE + Diffusion/DiT 或自回归模型
        ↓
推理加速
蒸馏 / KV cache / 并行解码 / 量化 / 剪枝
        ↓
评测
画质 + 动作跟随 + 几何一致 + 记忆 + 延迟
```

“用了 DiT”只回答生成主干是什么，不能说明它是否真能交互。两篇论文都用视频扩散，可能在动作接口、历史保存和推理速度上完全不同。

## 3. 瓶颈一：动作怎样进入模型

### 3.1 动作 token 与视觉 token 拼接

把 `W`、鼠标右转、机械臂位姿等编码成 token，与视频 token 放进同一序列。self-attention 可以直接学习“这个动作改变哪些区域”。

优点是统一、可扩展；缺点是模型可能把动作 token 当作弱提示，尤其历史图像很强时。

### 3.2 用动作缩放或平移视觉特征

动作网络输出 scale 与 shift，再调制视频 block：

\[
h'=\gamma(a)\odot h+\beta(a).
\]

- `h`：当前视觉特征；
- `a`：动作；
- `γ(a)`：动作决定的逐通道缩放；
- `β(a)`：动作决定的逐通道偏移；
- `⊙`：逐元素相乘。

它把动作写进每层特征，而不是只在序列开头出现一次。

### 3.3 相机轨迹与显式几何

六自由度相机位姿本来就有几何意义。3D 重建或渲染式方法可以先按新相机投影场景，再用生成模型补洞和修细节。几何控制通常更精确，但建立、更新 3D 表示更重。

### 3.4 文本控制

文字通常经 T5 等编码器后通过 cross-attention 注入。它适合“变成黄昏”“添加雨天”这类语义控制，不适合每 50 ms 精确描述摇杆位置。界面要匹配控制粒度。

### 一次给完与中途反复给的区别

```text
one-shot：提示词 + 轨迹 → 一次生成完整视频
multi-shot：历史 + 当前动作 → 下一段 → 接受新动作 → 再生成
```

后者不只是把前者循环调用。每轮输出都会成为下一轮输入，误差会累积；同时每轮都受延迟预算约束。

## 4. 瓶颈二：世界怎样记住走出画面的东西

### 4.1 直接把历史帧当条件

最自然的方案是：

\[
\epsilon_\theta(o_t\mid o_{\le t-1},a_{\le t-1}).
\]

即生成 `o_t` 时读取此前观察和动作。UniSim 把最近 4 帧与当前噪声样本沿通道拼接；许多后续方法会先用 VAE 或特征编码器压缩历史。

它像把最近几张照片摊在桌上。短期动作很清楚，但桌面大小固定：旧照片迟早被挤出去。

### 4.2 隐式 latent memory：存“过去看见过什么”

WorldMem 一类方法把历史视频 latent 连同位置、视角和时间戳存进 memory bank，需要时用 attention 检索。RELIC 把它分成：

- 未压缩的短期 KV cache，保存最近观察；
- 压缩的长期空间 memory，保存更远历史。

优点是动态物体和外观信息丰富；缺点是内存可能随视频长度增长，检索错了也会把无关画面带回来。

### 4.3 显式 3D memory：存“东西在空间哪里”

另一条路线保存点云、Surfel、3D Gaussian Splatting 或 mesh。Vmem 会根据当前视角与历史视角在 3D 中的重叠程度挑选记忆；Spmem 把静态点图与少量动态参考帧分开。

优点是回到旧地点时更容易保持几何；缺点是动态更新难：椅子被搬走后，旧 3D 地图可能仍把它画在原处。

### 4.4 混合记忆

MosaicMem 等方法把显式几何和隐式视频记忆结合：

| 记忆 | 擅长 | 容易失败 |
|---|---|---|
| 视频 latent | 外观、人物、动态 | 几何漂移，容量增长 |
| 显式 3D | 位置、视角重访 | 动态更新与遮挡 |
| 混合 | 同时保几何与外观 | 系统复杂，检索和同步都要维护 |

### 4.5 直接重建 3D 世界

Wonder 系列、WorldGen、HY-World 等不只把 3D 当记忆，而是直接生成可导航 3DGS、点云或 mesh，再渲染新视角。它把“视频续写”改成“维护场景再观察”，几何一致性更强，却需要额外重建、补洞、动态模拟和渲染模块。

## 5. 为什么 Forcing 方法能缓解越滚越歪

问题叫 exposure bias：

```text
训练：每一步都看真实历史 → 输入很干净
推理：每一步都看自己生成的历史 → 小错不断回流
```

论文把几种常见方案放在一起：

| 方法 | 训练时给过去帧什么 | 解决什么 | 仍有什么问题 |
|---|---|---|---|
| Teacher Forcing | 完全真实历史 | 训练稳定 | 与推理输入不一致 |
| Diffusion Forcing | 给历史加不同强度噪声 | 让模型见过“不完美历史” | 高斯噪声不像真实生成错误 |
| Self Forcing | 模型自己 rollout 的历史 | 输入最接近部署 | 误差可能无界积累，训练显存大 |
| LIVE | 向前 rollout 后再反向恢复 | 用 cycle consistency 约束漂移 | 多一次恢复路径 |
| Geometry Forcing | 对齐预训练几何特征 | 增强 3D 世界状态 | 受几何老师质量限制 |
| Context Forcing | 长上下文 memory student 自 rollout | 缩小长记忆 student 与短 teacher 的错位 | 训练链更复杂 |

这里的 “forcing” 不是强行把输出改成答案，而是决定**训练时模型拿什么当历史上下文**。

[[solaris-multiplayer-world-model]] 使用 Diffusion Forcing 把双向多人模型变成因果模型，再使用 Checkpointed Self Forcing 让 student 在自己的生成历史上学习。

## 6. 瓶颈三：记得越多，为什么反而越不听新动作

生成下一帧时，模型同时收到：

- 大量历史画面：“世界一直是这样”；
- 一个很小的当前动作：“现在立即向左转”。

历史越长，注意力中可重复的视觉证据越多。模型最安全的选择可能是延续旧运动，而不是响应新动作。于是长期一致性与即时控制形成拉扯。

可以把下一帧的依赖简化成：

\[
h_{\text{next}}=
\lambda\,h_{\text{history}}
+(1-\lambda)\,h_{\text{action}}.
\]

这不是论文的训练公式，只是解释旋钮：

- `λ` 太大：画面稳，但按键像没按；
- `λ` 太小：动作灵敏，但世界容易跳变。

Astra 给条件帧加随机噪声，主动削弱历史的统治力；Context Forcing 则让带长期 memory 的 student 在自生成轨迹上学习怎样同时利用历史与新动作。

所以不能只报告 FID。一个几乎复制上一帧的模型可能画质很稳，却完全不听控制。

## 7. 实时生成靠什么加速

### 7.1 少步蒸馏

把高质量、双向、多步 teacher 的能力迁移到因果、少步 student。例子：

- GameNGen 通过分布蒸馏报告 50 FPS，但论文指出画质有所下降；
- HY-GameCraft 蒸成 8 步 consistency model；
- Matrix-Game 3.0 用 multi-segment distillation 报告最高 40 FPS。

速度不能脱离模型大小、分辨率、GPU 与并行度横比。综述的 Table 6 也专门提醒这一点。

### 7.2 缓存

缓存有两种不同含义：

- [[kv-cache]]：保存过去 token 的 K/V，下一帧不重算整段历史；
- denoising-step cache：相邻去噪步复用中间 residual feature，例如 Yume 的逐层缓存策略。

HY-World 1.0 还把缓存与多 GPU 并行结合。缓存省计算，却占显存，并可能在模型状态快速变化时复用过时特征。

### 7.3 其他路线

综述还列出并行解码、关键帧重建、少步采样量化与模型剪枝。它们回答“怎样更快”，不自动回答“动作是否跟得准”或“长时间是否不漂”。

## 8. 评测不能只看一张画面好不好看

论文将数据与 benchmark 分成四类应用：

| 场景 | 最该测什么 | 常见缺口 |
|---|---|---|
| 开放世界探索 | 相机可控、3D 一致、重访记忆 | 早期视频 benchmark 只测画质 |
| 游戏引擎 | 动作空间、FPS、持续时长、记忆 | 不同硬件/分辨率速度不可直接比 |
| 具身机器人 | 动作跟随、物理交互、策略成功率 | 好看不等于能完成任务 |
| 自动驾驶 | 轨迹对齐、闭环安全、进度、舒适性 | open-loop 图像指标不代表规划可靠 |

### 指标各自回答什么

- **FID / FVD**：生成图像或视频分布像不像真实数据；不直接懂动作。
- **ADE**：生成轨迹与条件轨迹对应时间点的平均距离。
- **DTW**：允许时间非线性对齐后比较两条路径形状。
- **action following**：动作是否在画面中产生预期结果。
- **3D consistency**：换视角后结构是否保持。
- **memory consistency**：离开再回来，旧对象与布局是否恢复。
- **FPS / latency**：多快反馈；必须连同分辨率、硬件和步数一起读。
- **下游成功率**：机器人或驾驶策略是否真完成任务，是更接近用途的闭环指标。

### 不要把综述表格误读成统一排行榜

表里的数字来自不同原论文，训练数据、硬件、分辨率、任务和评测协议不同。例如 Genie 报 360p、2 秒，Matrix-Game 3.0 报 720p、40 FPS；不能仅凭 FPS 判断谁更强。综述提供的是能力地图，不是受控实验。

## 9. 研究趋势：模型正在沿三条轴扩展

### 专家模型 → 通用模型

从只会单个游戏或单一驾驶域，走向能处理多场景、多任务的 foundation world model。通用性扩大，动作语义一致性与数据清洗也更难。

### 静态 / 单智能体 → 动态 / 多智能体

世界不只是相机在静态场景中移动，还包含物体变化、其他智能体行动和相互影响。[[solaris-multiplayer-world-model]] 就要求同一动作同步影响两位玩家视角。

### 单一视觉 → 多感官交互

输入扩展到文字、声音、力、遥操作与物理参数。不同模态更新频率不同：音频需要密集连续反馈，文字适合稀疏高层指令，机器人控制则需要精确数值。

## 10. 五个真正没解决的问题

### 10.1 大规模动作标注

真实机器人与驾驶动作贵、危险、域相关。无标注视频可以学 latent action，但 latent 可能把相机运动、外观和数据偏差混在一起，不一定对应可执行控制。

### 10.2 反事实与现实因果

模型可能生成“看起来合理”的未来，却没学会“正因为执行这个动作，所以发生这个变化”。安全场景需要干预数据、对象级状态、动力学先验和真实闭环验证。

### 10.3 小时级长期一致

分钟级 rollout 已开始出现，但小时级仍会漂移、物体消失、结构变化。可能需要分层记忆：短期保存动作细节，长期保存地点、对象身份和全局世界状态。

### 10.4 物理理解

视频扩散擅长模仿“像物理”的运动，不等于知道刚体、软体、力和碰撞。显式模拟器、物理粒子、几何基础模型和生成后验证可能需要进入闭环。

### 10.5 真正可直接操作的界面

用户不应每次靠长 prompt 描述“把那个杯子拖到这里”。综述借用 direct manipulation 原则：对象和动作可见、操作快速可逆、结果立即反馈。未来接口需要把用户看到的物体直接连接到模型内部可编辑表示。

## 11. 用三问读任何新世界模型

看到一篇新论文，先别数模型参数，按下面顺序问：

1. **动作怎样进入？**是一开始给一次，还是每帧可更新；是文字、token、位姿还是渲染控制？
2. **历史保存在哪里？**最近帧、KV cache、latent memory、显式 3D，还是根本没有？
3. **速度怎样来的？**少步蒸馏、缓存、并行、量化；报告是否同时给硬件、分辨率和延迟？

然后再看评测是否覆盖动作、记忆、几何、画质和闭环任务。若只给 FID/FVD，它更像可控视频生成器的质量报告，还不足以证明“世界模型”。
