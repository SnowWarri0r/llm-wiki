# MeshFlow · 把"谁连谁"改写成连续数,整张网格一次并行长出来

## 一句话
生成 3D 网格最难的不是"顶点摆哪儿",而是"哪两个顶点该连一条边"——前者是连续坐标、后者是离散开关。MeshFlow 给每个顶点挂一枚连续的"连边磁铁",让扩散模型只生成连续量,最后一步才把磁铁的吸力切成 0/1 的边;再配上 VAE 压缩,整张网格一次并行生成,不用一根一根边地吐。

## 它要解决的痛点
一张三角网格 = 一份顶点表 + 一份面表。顶点 `(0.31,−0.42,0.08)` 是连续数,可以平滑地加噪声、去噪声——扩散模型最擅长这个。可面 `(17,42,91)` 是三个整数索引,把 `42` 改成 `43` 没有"改一半"的中间态。**连续坐标好办,离散连接才是硬骨头**,几条老路都各有别扭:

- **先做隐式表面、再抽网格**(SDF + Marching Cubes):形状能出,但三角面又密又糊,锐边被磨平,还常要求封闭表面,美术师拿到手很难改。
- **把坐标和面全切成离散 token、自回归逐个吐**:拓扑清楚,但坐标被量化到每轴 128 档(细节发抖),而且网格越大、串行解码越慢,长网格还容易提前"说完了"。

MeshFlow 换个思路:**别让扩散模型碰离散的东西**。把"有没有边"也改写成连续数,让模型全程只在连续空间里玩,离散化推到最后一步再做。

## 核心贡献
- **连续连接表示**([[continuous-mesh-connectivity]]):每个顶点带三样连续量——坐标、外法线、一枚"连边磁铁"(边嵌入)。两枚磁铁的吸力分数过阈值就连一条边,三条边围成一圈就补一个三角面。离散拓扑被这枚磁铁"翻译"成了连续向量。
- **MeshVAE 压缩**([[mesh-token-merge]]):把 N 个顶点压成约 N/4 个 latent。压法叫 TokenMerge——不是丢点,而是把相邻几个顶点的通道拼一块、再让短 query 回头用 cross-attention 看全部顶点,所以"变短"不等于"看不见"。
- **并行生成**([[flow-matching]] + [[diffusion-transformer]]):在压好的 latent 上跑整流流 DiT,一次采样步同时更新整串 latent(默认 28 步),而不是一根边一根边地自回归。
- **点云指路**([[voxel-rope-conditioning]]):把条件点云先落进 32³ 的粗体素格,再用 3D RoPE 把"点在哪块空间"写进注意力——只记大致位置、不记采样细节,缩小训练和推理的口音差。

## 方法主线

### 1. 先把网格改写成"一张连续的顶点表"
不直接生成面,而是给每个顶点一条连续向量 `(v, n, h)`:`v` 是坐标、`n` 是外法线(指向表面外)、`h` 是连边磁铁。扩散模型只管生成这些连续量;解码最后一步才把任意两枚磁铁的吸力算出来,过线的连成边,三条边成环补成一个三角面。法线在这里当"朝向裁判":几何算出的面法线要是跟三个顶点法线的平均方向相反,就把面的两个顶点交换一下,把正反面翻回来。好处是不挑输入——不要求封闭流形;代价是只做三角面。

### 2. MeshVAE 把顶点表压短
逐顶点跑 DiT 太贵,先用 VAE 压。编码器给每个顶点拼上坐标、法线、邻居坐标(都过一遍 Fourier 编码),然后 **TokenMerge**:把相邻 4 个顶点 token 的通道拼成一个宽 token、再用一层线性压回原宽度,序列就短了 4 倍。关键是这批短 token 随后用 **cross-attention 回看全部原始顶点**——它们不是空白工位,而是抱着原料上桌,所以压短没有立刻丢掉信息。解码端做反向的 TokenSplit 把序列摊回来。

### 3. 从一堆候选边拼回三角面
解码后先按 mask 删掉补齐用的空位,再两两算磁铁吸力。对每条边 `(i,j)` 找它俩的共同邻居 `k`,如果 `ij`、`jk`、`ki` 三条边都在,就收下三角面 `(i,j,k)`,最后用法线校正朝向。麻烦点:三条"错误边"要是恰好凑成一个环,也会凭空长出一个错误面。

### 4. 整流流 DiT 并行生成 latent
训练时把干净 latent `x₀` 和高斯噪声 `ε` 用一条直线连起来:中间点 `x(t)=(1−t)x₀+tε`,这条直线的速度恒定是 `ε−x₀`。DiT 看着带噪点、时间 `t` 和点云条件,去回归这个速度。推理反过来:从纯噪声出发,顺着预测速度一步步走回 `t=0`,再交给 VAE 解码成网格。这里的"并行"指**每一步同时更新整串 latent**,砍掉了自回归那条越来越长的串行链;采样轮数固定(默认 28 步),不随网格变大而变长。

### 5. 点云怎么给 DiT 指路
难点是训练看的是网格顶点、推理看的是表面均匀采样点,两者分布不一样(美术师爱在锐边多放顶点,均匀采样只按面积撒)。直接把精确坐标喂进去,模型会记住"顶点怎么排",一到推理就"换了口音"。解法:先把点落进 32³ 的粗体素格,同格的点共用一个位置编号,再用 3D RoPE 把这个粗位置写进注意力。模型只抓"点在哪块空间",不纠结采样细节,训练推理的差就小了;代价是条件几何变粗。

## 训练与结果(挑关键的)
- **数据**:约 60 万个专业美术师做的私有网格,归一化到 `[−1,1]³`,不量化坐标,过滤掉最大顶点度数 >50 的。
- **规模**:MeshVAE 233M(32 张 H100 训 3 天);Flow DiT 约 4-9 亿参数(64 张 H100 训 10 天)。
- **压得狠**:表示长度只占别人的约 1.4%(压缩比 0.014,对比 EdgeRunner 0.47、TreeMeshGPT 0.22),重建质量还接近最优(CD×100 = 1.29,略逊 EdgeRunner 1.21、好于 TreeMeshGPT 1.63)。所以准确说法是"用短得多的表示做到接近最优的重建",不是"重建误差全场最低"。
- **又快又好**:Toys4K 上 CD/HD = 2.33 / 4.23,好过 FastMesh 的 4.09 / 10.32;单物体约 1.2 秒。
- **消融的亮点**:TokenMerge 起手的 query 带原始内容,边 F1 从随机 query 的 49 / 最远点采样的 60 直接拉到 99.78——说明"query 从哪儿起步"对压缩质量极关键。

## 我的批注 / 疑问
- 最值钱的不是"又一个 flow DiT",而是**先发明了一种扩散模型吃得下的网格**:坐标管形状、边嵌入管连接、法线管朝向。表示一理顺,VAE 压缩和 Flow 并行才顺理成章。
- "18× 加速"要留个心眼:那是**单物体**口径(FastMesh ~21s vs MeshFlow ~1.2s);主表按 batch 均时口径只有约 **2.2×**。两个数都真,但引用时得把分母写清楚。
- "非自回归 = 对网格大小线性"说得太满:它把**串行深度**从输出长度降成固定采样步数(这是真优势),但标准 DiT 每层仍是全注意力,单层计算对 token 数还是二次。稳妥理解是"生成轮数不随输出变长",不是"总计算严格线性"。
- **复现坑(codex 查证)**:论文、补充材料、公开代码有三处对不上——① 连边阈值主文写 `d≤τ`,代码实际是 `d>τ`(代码里的 `d` 是"越大越像边"的带符号分数,不是普通距离);② DiT 规模主文 18 层/427M、补充材料 16 层/895M,两套冲突;③ 补洞默认关闭。复现时把论文版本 `v2` + 代码提交 `55f56f6` + gated 模型配置一起记下来。
- 论文最诚实的一处:主动不报 Normal Consistency——模型直接预测法线,拿这项高分不能证明网格更好;可惜拓扑质量至今没有可靠的量化指标(CD/HD 都看不见翻面和洞)。
