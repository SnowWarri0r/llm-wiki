2026-07-17conceptFlashAttention · 快的不是少算，而是少把中间结果写回显存普通注意力常拆成三次 GPU 计算：先算分数矩阵，再做 softmax，最后乘 `V`。三次计算之间，前一步的结果要先写回 HBM，后一步再读出来。最占地方的恰好是两个 `N × N` 中间矩阵。 2026-07-17conceptGeneralized Causal Attention · 一条多模态序列里，精确规定谁能看谁普通语言模型只允许后面的 token 看前面的 token；Bagel 把这条规则扩展到文字、输入图和待生成图：文字仍按顺序读，同一张图内部可以双向互看，后续内容可以读取已经完成的条件，但不能偷看尚未生成的答案。 2026-07-17conceptMixture-of-Transformers · MoT · 按 token 类型分开的两套 Transformer 权重每个 Transformer 层有两套完整权重；token 按类型只走其中一套，但两类 token 仍在同一条序列里互相注意。 2026-07-17conceptUnified Multimodal Generation · 统一多模态生成不再给检测、分割、深度和 3D 各接一颗专用 head，而是把答案翻译成底座本来就会生成的**文本、图像或文本+图像**，再按确定协议还原成框、mask、深度或 3D 坐标。 2026-07-16bookTrading in the Zone · 交易心理分析交易输赢不是分析题，是心理题——**再准的分析也解决不了"看得懂却做不到"，真正的门槛是学会在"下一步不确定"的处境里，不害怕、不轻率地按概率一次次执行**。 2026-07-16conceptActivation Checkpointing · 少存中间结果，反向时重新算前向时少记几份“草稿纸”，反向传播需要时再重算；省的是显存，花的是计算。 2026-07-16concept分布式训练并行 · 模型、数据和长序列分别怎么切显存不够时不要只说“多加几张卡”：参数、矩阵乘和 token 是三种不同的东西，要用三种切法。 2026-07-16conceptGPU 集群调度 · 为什么 128 卡任务不能先给 120 卡分布式训练要整组资源一起到位；只启动一半，已经拿到卡的进程也只能空等。 2026-07-16conceptGPU 互联与集合通信 · 多卡训练为什么常卡在“搬”而不是“算”NVLink、PCIe、InfiniBand 是路；NCCL 是组织车队的通信库；all-gather、reduce-scatter 是具体搬法。 2026-07-16conceptGPU Kernel 与编译 · 为什么同一个公式能快很多Kernel 是 GPU 一次执行的一小段程序；编译优化的目标，是少启动、少搬数据、选对专用实现。 2026-07-16concept大规模数据管线 · 素材、元数据和任务进度怎么不打架大文件放数据层，小状态放元数据层；worker 靠租约或行锁领活，处理成功后再原子提交。 2026-07-16concept模型服务栈 · PyTorch、TensorRT、vLLM 和 vLLM-Omni 各管什么模型服务不是换一个“更快框架”就结束：运行时、请求调度、跨卡并行、缓存和解码各管一层。 2026-07-16concept训练 Checkpoint 与恢复 · 大训练默认会崩，关键是少丢进度长训练不赌“永不失败”，而是让存档足够频繁、写盘不挡训练、重启后能接着跑。 2026-07-14paperMeshFlow · 把"谁连谁"改写成连续数,整张网格一次并行长出来生成 3D 网格最难的不是"顶点摆哪儿",而是"哪两个顶点该连一条边"——前者是连续坐标、后者是离散开关。MeshFlow 给每个顶点挂一枚连续的"连边磁铁",让扩散模型只生成连续量,最后一步才把磁铁的吸力切成 0/1 的边;再配上 VAE 压缩,整张网格一次并行生成,不用一根一根边地吐。
