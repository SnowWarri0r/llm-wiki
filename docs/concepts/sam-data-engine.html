Concept

SAM Data Engine · 数据引擎（模型在环的标注飞轮）

一句话

模型帮人标注、标注再训模型、模型越强标得越多——三阶段滚雪球，从零滚出 11 亿掩码。

直觉

先有鸡还是先有蛋：要训分割基础模型得有海量掩码，可像素级标注贵得离谱，没数据就训不出好模型、没好模型标注又快不起来。

SAM 的破法是飞轮：让当前模型先去标，人只补它标不好的；补出来的新数据又拿去把模型训得更强；模型更强 → 标得更准更快 → 人补得更少……越转越快，最后人几乎可以撒手，模型全自动标完剩下的。这跟 whisper 的弱监督、unlimited-ocr 用 PaddleOCR 自动标注是同一个母题：大规模数据靠模型自己造，人只在早期推飞轮。

它能转起来的前提是 promptable-segmentation 的轻量解码器——换 prompt 只要 ~50ms，标注员才能实时点几下就出掩码，交互成本低到飞轮转得动。架构和数据互为因果。

怎么做的

阶段1 · 辅助手动:  人用 SAM 交互工具标, SAM 给初始掩码人来修
阶段2 · 半自动:    SAM 自动标出它有把握的, 人只补漏掉的(逼数据更多样)
阶段3 · 全自动:    32×32 网格点当 prompt 喂给 SAM, NMS 去重 → 全自动产掩码

每过一阶段，模型被新数据喂得更强，下一阶段人就更省力。

数字例子

三阶段的产出和"人退场"的过程，数字很能说明问题：

阶段1 辅助手动:  4.3M 掩码 / 120k 图
                单掩码标注时间 34s → 14s(模型变强,人修得快)
                每图掩码数    20 → 44
阶段2 半自动:    +5.9M 掩码 / 180k 图
                每图掩码数    44 → 72(自动补上不显眼的)
阶段3 全自动:    32×32 网格点扫全部 11M 图 → 1.1B 掩码
─────────────────────────────────────────────
SA-1B 总计: 11M 图 / 1.1B 掩码, 其中 99.1% 全自动生成
            比已有任何分割数据集多 400×; 94% 掩码与专业修正 IoU>90%

✓ 自检：人手标的只有阶段1+2 的约 10M 掩码（不到 1%），剩下 99% 是模型自动产的——但没有前两阶段把模型喂到够强，阶段3 的全自动根本标不准。飞轮头几圈靠人推，后面才自转。
