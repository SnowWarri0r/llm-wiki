Concept

NAR 掩码并行生成 · 像完形填空一样"一次性填声"

一句话

不逐帧自回归吐 token,而是先把要生成的码全部盖住(掩码),一次性并行猜出全部、再迭代几步把没把握的重猜——生成语音从"逐帧慢慢来"变成"整段一起填",快很多。

直觉 · 自回归 vs 非自回归

  • 自回归(AR):一帧一帧生成,第 n 帧要等第 n−1 帧算完。200 帧就要 200 步,长音频天然慢、延迟高。
  • 非自回归(NAR):所有帧同时预测。像 BERT 完形填空——把整句盖住,一次性并行填空,再看哪几个填得没把握、盖回去重填,几轮就收敛。步数固定(几步),跟长度基本无关 → 快。

关键区别:AR 的成本 ∝ 序列长度;NAR 的成本 ∝ 迭代轮数(常数)。所以 NAR 在长音频、低延迟场景赢。代价是要迭代精修才能补回 AR 那种逐帧一致性。

怎么做的 · 掩码 → 并行猜 → 迭代精修

初始:  [MASK][MASK][MASK] … [MASK]     全盖住
第1轮:  并行预测全部码 → 保留置信度高的,低的重新盖回
第2轮:  只对盖回的位置再并行预测 …
…几轮后收敛,所有位置定稿

训练用随机掩码:盖住一部分码让模型学着从上下文补全(OmniVoice 用"全码本随机掩码",并从预训练 LLM 初始化)。这就是 SoundStorm / MaskGCT / OmniVoice 一脉,viitorvoice 的声学解码器属于这家。

数字例子 · 步数对比

生成 200 帧声学码:
AR:   200 步(逐帧,每步等前一帧)
NAR:  ~8 步(每步并行填全部 200 帧的剩余位置)
      → 步数从 200 降到 ~8,且每步内部并行

✓ 只要迭代轮数(~8)远小于序列长度(200),NAR 就快一个数量级。TTS 里语义骨架已由 LLM 定好,声学细节容错高,正适合这种"并行填、几步收敛"。
