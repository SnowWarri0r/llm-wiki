Go GC ·
从乱跳到顺着扫.

Go Runtime Team · Michael Knyszek 等 · Green Tea GC · Go 1.26 · 2026

Go 的 GC 这些年的主线是"让标记不再卡在等内存"。Go 1.26 默认开的 Green Tea 把工作单元从"对象"换成"8 KiB 页"，把指针追逐的随机访存改成按页顺序扫 —— GC-heavy 程序 GC overhead 降 10–40%，还不用改一行代码。

TL;DR

Go GC = 三色并发 mark-sweep1 + write barrier3，标记跟你的程序并发跑，几乎不长时间 STW2。触发节奏靠 GOGC6（默认堆涨到 2× live 触发）和 GOMEMLIMIT7（硬上限）控。老的标记是 graph flood11 —— 顺着指针在堆里乱跳，mark 时间里 ≥35% 纯 stall 在等内存。Go 1.26 默认开的 Green Tea12 把工作单元换成 8 KiB 页10：每页攒一批待扫对象、按地址顺序一次扫完，随机访存变顺序访存；现代 CPU 上再用 AVX-512 几条指令扫一整页。结果：GC CPU 降 10–40%，语义不变。

§ 01底座 · 三色并发 mark-sweep

GC 要回答一个问题：堆上哪些对象还活着，哪些是垃圾。Go 的答法是 mark-sweep2 —— 先 mark（从根出发，把所有能到达的对象标记成"活"），再 sweep（没被标到的就是垃圾，回收）。

判断"活"的起点是 GC roots5：每个 goroutine 的栈、全局变量这些"程序直接拿在手里"的指针。从 roots 能顺着指针链到达的，全是活的。

标记过程用 三色不变式1 来跟踪进度。每个对象三种颜色之一：

对象在堆上不是随便放的：Go 把同一个 size class9 的对象凑在同一段连续内存（span）里，一个 span 是若干 page。这条"同类对象住一起"的事实，后面 Green Tea 会狠狠用上。

§ 02并发的代价 · write barrier

如果 GC 肯让世界停下来（STW）慢慢标记，三色就够了。但 STW 几百毫秒在线上不可接受，所以 Go 的标记是跟你的程序并发跑的 —— GC 在标记的同时，mutator4（你的业务 goroutine）还在改指针。

这就出事了。考虑这个时序：一个对象已经被标黑（GC 认为它扫完了，不会再看），这时 mutator 把它的某个字段改成指向一个白对象，同时那个白对象原本唯一的灰色入口被删掉。结果：白对象只剩黑对象指它，而黑对象不会再被扫 —— 它永远不会被标记，最后被当垃圾回收，留下悬空指针。

write barrier3 就是编译器在每条"写指针"操作前后塞的一小段代码：标记进行时，凡是改指针就顺手把相关对象染灰，保证不破坏三色不变式。代价是每次指针写都多几条指令 —— 这也是为什么 Go 的 GC 只在标记期开启 barrier，标记一结束就关掉。

§ 03什么时候触发 · GOGC 与 GOMEMLIMIT

GC 跑得太勤 → 浪费 CPU；跑得太懒 → 堆涨爆内存。这个节奏由 pacing8 控制，两个旋钮：

GOGC6（默认 100）：上次 GC 后存活的堆是 live，那么堆涨到 live × (1 + GOGC/100) 就触发下一次。GOGC=100 → 涨到 2× live 触发。调大（GOGC=200）→ GC 更少、内存换 CPU；调小 → 反之。

GOMEMLIMIT7（1.19+）：一个软硬上限。即使按 GOGC 还没到触发点，只要堆逼近 GOMEMLIMIT，GC 就提前触发，把堆压在限额下 —— 用来防 OOM。代价是逼近限额时 GC 会变得很频繁（GC CPU 飙升），所以它是"别 OOM"的安全网，不是日常调优主力。

到这里 Go GC 的"骨架"就齐了：三色并发 mark-sweep + write barrier 保证正确，GOGC/GOMEMLIMIT 控节奏。剩下的故事全是怎么把 mark 这一步做快 —— 这正是 Green Tea 干的事。

§ 04痛点 · graph flood 在内存里乱跳

老的标记叫 graph flood11：从灰色队列取一个对象 → 扫它的指针 → 把引用对象染灰塞回队列 → 取下一个。问题是这本质是指针追逐 —— 对象之间互相指，但它们在内存里八竿子打不着，你顺着指针跳，几乎每跳一次就是一次 cache miss + TLB miss。

官方实测：mark 的时间里，至少 35% 是纯粹 stall 在等堆内存。而且这个问题随堆变大、对象变小变碎而恶化，加 CPU 核也救不了 —— 所有核都堵在等内存。mark 从 CPU-bound 退化成 memory-latency-bound。

用开车打比方：graph flood 像在城市街道乱开 —— 看不见下个路口，红绿灯随机，走走停停；Green Tea 像上了高速 —— 路直、能看远、CPU 可以放心预取。对象一个没少、图一点没变，变的只是你按什么顺序去碰内存。

§ 05Green Tea · 把工作单元从对象换成页

Green Tea12 的核心想法官方自己说"简单得离谱"：以页为单位干活，不以对象为单位。 Go 的一个 page 是 8 KiB10 对齐内存块，同 size class 的对象住同一页。具体三处变化：

• 工作队列里排的是页，不是对象
• 一页待扫的对象用 seen / scanned 两个 bitmap 在页本地记录，不再全堆维护一个巨大灰色队列
• 队列从栈（后进先出）改成 FIFO（先进先出） —— 让页在队列里多待一会，把"被指到"的对象攒一批，再一次性按地址顺序扫完

把这些合起来：随机访存变顺序访存，work list 从全堆对象缩成页、竞争和 stall 都降下来，再叠加向量指令。官方 benchmark：不算向量就有 10–40% 的 GC CPU 下降，modal（最常见）约 10%；现代 CPU 加上 AVX-512 再多 ~10%。Google 内部大规模上线，结果一致。

一个反直觉的发现

官方说：哪怕一次只扫一页里 2% 的对象，都已经比 graph flood 快。这说明赢点根本不在"批量摊销开销"，而在 locality 本身 —— 顺序碰内存这件事的价值，比你直觉以为的大得多。

§ 06实战 · 升 Go 1.26 意味着什么

Green Tea 在 Go 1.25 是实验开关（GOEXPERIMENT=greenteagc），Go 1.26 默认开启。它是纯 locality 优化 —— GC 语义、GOGC/GOMEMLIMIT 行为全不变，你不用改任何代码。唯一能观察到的就是 GC CPU% 掉。

• 谁受益最大：GC-heavy、小对象 churn 大的服务 —— JSON 重的请求处理、大量短命小分配。GC 本来就轻的几乎无感。
• 怎么量：升级前后用 runtime/metrics（/gc/cycles/total、GC CPU 那几项）或 pprof 对比 GC CPU 占比。
• 现代 CPU 额外加成：跑在 Intel Ice Lake / AMD Zen 4 及更新的机器上，自动吃到 AVX-512 那 ~10%。
• 怎么关：GOEXPERIMENT=nogreenteagc（build 时）。官方明确说：如果因为性能或行为问题关掉它，请提 issue。

典型受益场景：请求里狂分配小对象、JSON 序列化重、GC 压力不小的 Go 服务 —— 升 1.26 后值得拉一下 GC CPU 的前后对比，大概率能看到掉一截，等于白捡的吞吐。

更大的图景：这是 Go GC 演化主线的又一步 —— 从"能并发、低延迟"（三色 + write barrier 时代）走到"对现代内存层级和 SIMD 友好"。GC 的瓶颈早就从"算得快不快"变成"等内存等多久"，Green Tea 是第一个正面冲着访存延迟去设计的 Go GC。

引用 / 术语表

  1. 三色标记 · tri-color marking · 跟踪标记进度的方法。对象分白（没被标到）/ 灰（被指到但还没扫它指谁）/ 黑（自己和指向的都处理完）。核心不变式：黑对象绝不直接指向白对象 —— 这是并发标记不漏活对象的根基。
  2. mark-sweep · STW · GC 两阶段：mark 从根出发标记所有可达对象，sweep 回收没被标到的。STW（stop-the-world）= 暂停所有业务 goroutine；Go 把 STW 压到极短（亚毫秒），标记主体跟程序并发跑。
  3. write barrier · 写屏障 · 编译器在每条"写指针"操作处插的一小段代码。并发标记时凡改指针就顺手染灰相关对象，保证不破坏三色不变式、不漏标。代价是指针写多几条指令，所以只在标记期开启。
  4. mutator · GC 视角下的"你的业务代码" —— 那些在 GC 标记的同时还在分配对象、改指针的 goroutine。GC 和 mutator 并发跑，正是 write barrier 存在的原因。
  5. GC roots · 根集合 · 标记的起点：每个 goroutine 的栈、全局变量等"程序直接拿在手里"的指针。从 roots 顺指针能到达的全是活对象，到不了的就是垃圾。
  6. GOGC · 控 GC 触发频率的主旋钮（默认 100）。上次 GC 后存活堆是 live，堆涨到 live×(1+GOGC/100) 触发下次。GOGC=100 → 涨到 2× live 触发。调大省 CPU 费内存，调小反之。
  7. GOMEMLIMIT · 内存软上限（Go 1.19+）。堆逼近这个值时 GC 提前触发把堆压在限额下，用来防 OOM。代价是逼近时 GC 很频繁、GC CPU 飙升 —— 它是安全网不是日常调优主力。
  8. GC pacing · 配速 · runtime 决定"何时触发、标记多快"的控制逻辑，目标是在 GOGC 设定的内存预算内，让并发标记刚好在堆撞到触发点时完成，既不早停也不超支。
  9. size class · span · Go 把对象按大小归到约 70 个 size class，同 class 的对象集中放在同一段连续内存（span，由若干 page 组成）。"同类对象住一起"是 Green Tea 能按页批量扫的前提。
  10. page · 8 KiB · Go runtime 的内存管理基本块，8 KiB 对齐。Green Tea 把 GC 工作单元从单个对象提升到 page —— 一页里同 size class 的对象一起扫。
  11. graph flood · 图洪泛 · 老的标记算法：逐个对象从灰色队列取出、扫指针、染灰、再取下一个。本质是指针追逐，在堆里随机乱跳，cache/TLB miss 极多 —— mark 时间里 ≥35% 纯 stall 在等内存。
  12. Green Tea GC · Go 1.26 默认 GC。工作单元从对象换成 8 KiB page，用 seen/scanned 两个 bitmap 在页本地记录，FIFO 队列攒一批再按地址顺序扫 → 随机访存变顺序访存。现代 CPU 用 AVX-512 GFNI（VGF2P8AFFINEQB）几条指令扫整页。GC CPU 降 10–40%，语义不变。
