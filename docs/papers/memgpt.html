FIELD NOTES · AGENT MEMORY · OS

MemGPT ·
LLM 自己当 OS.

Charles Packer · Sarah Wooders · et al. · UC Berkeley · 2023

把 LLM 的 context window 当 RAM, 外部存储当硬盘, 让 LLM 自己通过 function call 管理自己的记忆。OS 级别的记忆调度套在 agent 身上。

TL;DR

LLM 的 context window 装不下所有对话历史和外部知识。Generative Agents 那套用外部代码编排记忆调度1，MemGPT 反过来 —— **让 LLM 自己当 OS**2，通过 function call 决定哪些记忆留 context、哪些 page out 到外部、什么时候 page in 回来3。三层记忆 (core / recall / archival)4 对应 OS 的 register / RAM / disk。这是给 LLM agent 的 OS 抽象。

§ 01痛点 · context 装不下，谁来管？

LLM 推理是无状态的 —— 每次只看 prompt 里塞了什么。但实际 agent 场景里，你需要 LLM "记得"：

• 用户上周说过他喜欢吃辣
• 三天前的对话里讨论过某个项目
• 用户上传的那份 50 页文档
• agent 自己的人设和长期目标

这些加起来可能几十万 token，远超 context window (GPT-4 是 128K, 大多数模型还更小)。谁来决定什么时候把什么塞进 context、什么时候把什么挪出去？

Generative Agents 的答案是：**外部代码编排** —— 每一步都跑检索 + 评分 + 排序，挑 top-K 塞进 prompt。LLM 是被动接收者。

MemGPT 的答案完全相反：**让 LLM 自己当大脑**。LLM 知道自己 context 快满了，知道哪条消息可以扔掉，知道什么时候该去翻历史。所有这些决策都通过 function calling5 来执行 —— 就像一个会自己管自己内存的程序。

§ 02核心隐喻 · OS 的虚拟内存

MemGPT 这套设计完全从 OS 的虚拟内存机制借来。先回忆一下 OS 怎么管内存的6 ——

程序需要的数据可能远超 RAM 容量。OS 的解决方案：

1) 把数据切成 page (页)
2) 当前在用的页放 RAM 里 (快但贵)
3) 不用的页换到硬盘 (慢但便宜)
4) 程序访问硬盘上的页时触发 page fault，OS 把那页换回 RAM
5) 程序感觉自己有无限内存，实际上 RAM 在后台不停换页

实际系统里 两套常常被混用：

• 反思 / 长期总结 走 Generative Agents 思路 —— 用外部 cron job 定期摘要 (LLM 自己难以准确判断"现在该总结了")

§ 06实际场景 · 长对话不丢人设

MemGPT 论文里展示的核心场景是 "超长对话保持人设和记忆一致性"9。

典型测试：让 agent 跟用户聊 100+ 轮，中间隔几天甚至几周。考察的是：

• Agent 是否还记得用户的关键信息 (姓名、偏好、上次提到的事)
• Agent 的人设是否漂移 (说话风格、立场)
• Agent 能否引用很久之前的对话内容

普通 LLM 在这种场景下会失败 —— context 装不下 100 轮对话，简单截断会丢关键信息。MemGPT 把 100 轮对话压缩成 "core memory 里的几条 fact + recall memory 里完整历史"，关键时刻 LLM 主动检索。

# 第 50 轮对话中
User: "你还记得我跟你说过我女儿叫什么吗？"

LLM 推理:
  → 我的 core memory 里只有"用户有一个女儿"这条 fact
  → 但具体名字不在 core 里
  → 调用 conversation_search("女儿的名字")

[framework executes search → returns: "用户在第 8 轮提过, 女儿叫小雨"]

LLM 输出: "记得啊, 你女儿叫小雨。最近她怎么样？"

  → 同时调用 core_memory_replace(
      label="user.daughter_name",
      content="小雨"
    )
  → 这次更新后, 下次问就不用再查了

§ 07历史定位 · 后来变成了 Letta

MemGPT 团队后来把论文产品化成了 Letta (letta.com)10 —— 开源 agent 框架，把记忆 + 工具调用 + 人设管理打包。

MemGPT 这套 OS 抽象在 2024-2025 年的几个影响：

• 函数调用 + 记忆 成了主流 agent 框架的默认配置 (LangChain / LlamaIndex / Letta / Mastra 等)
• 三层记忆 (核心 / 对话 / 长期) 是绝大多数 AI companion / chatbot 产品的标配
• "LLM 自调度" 思路被进一步扩展到 ReAct、Tree of Thoughts、AutoGPT、Devin 等更复杂的 agent 系统

MemGPT 跟 Generative Agents 一起，定义了"2023 年的 agent 记忆架构"。所有后来的 agent 系统都在它们的基础上往前走。

引用 / 术语表

  1. 外部编排 · Generative Agents 那套架构里，是外面的代码决定"现在该检索 / 现在该反思"，LLM 只负责执行被喂进来的 prompt。LLM 是被动的执行单元。详见 memory-stream。
  2. LLM 自己当 OS · MemGPT 的核心 thesis —— 把 LLM 的角色从"会思考的工人"升级到"会思考 + 会管自己 context 的程序"。借鉴 OS 的虚拟内存机制。详见 virtual-context-management。
  3. Function calling 调度记忆 · LLM 不直接读外部数据库，是通过调用类似 search_memory(query) / archive_messages(ids) 这种函数来间接操作。函数执行结果作为下一步 context 的一部分。
  4. 三层记忆 · core / recall / archival · MemGPT 的核心数据结构。Core 永远在 context (类比 CPU 寄存器)、Recall 是对话历史的完整持久化 (类比 SSD)、Archival 是任意外部信息 (类比外接硬盘)。三层各自有专门的函数操作。
  5. Function calling · 函数调用 · LLM 在 2023 年获得的关键能力 —— 不只输出文字，还能输出结构化的"函数名 + 参数"，由框架执行后返回结果。整个 MemGPT 建立在 GPT-3.5/4 的 function calling 能力之上。
  6. 操作系统虚拟内存 · OS 经典机制：用 page in / page out 让程序"感觉拥有"远超 RAM 容量的内存。MemGPT 完整借鉴了这套思路 —— page fault、page out、page in 三个核心动作都对应到 LLM 的具体函数调用。
  7. Context 90% 阈值 · 当 context 接近上限时，MemGPT 框架自动注入一条 "你 context 快满了" 的系统消息。LLM 看到后主动决定怎么 archive。这是整个系统能跑起来的关键触发点。
  8. 外部编排 vs LLM 自治 · 两种 agent 记忆调度路线的根本分野。前者把 LLM 当工人 (Generative Agents)，后者把 LLM 当 OS (MemGPT)。两种都对，看 LLM 能力强弱 + 工程容忍度。
  9. 超长对话一致性 · MemGPT 论文的核心 benchmark 场景。让 agent 跟用户聊 100+ 轮（甚至跨天跨周），考察是否保持人设、是否记得关键 fact、是否能引用很久之前的对话。普通 LLM 这种场景下崩；MemGPT 通过 core memory + recall memory 检索撑过去。
  10. Letta · 论文的产品化 · MemGPT 团队 (UC Berkeley) 后来把论文产品化成 Letta，开源框架，把"function call 调度记忆"打包成开箱即用的服务。letta.com
