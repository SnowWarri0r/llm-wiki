Baidu Inc. · Unlimited OCR Technical Report · 2026-06-22

别的 OCR 一页一页 for-loop 跑、每页重置记忆。Unlimited OCR 把解码器注意力全换成 R-SWA，KV cache 恒定不增——几十页一次前向读到底。

TL;DR

拿 LLM 当 OCR 解码器有个硬伤：输出越长，KV cache 越堆越大、生成越来越慢，所以现有模型只能一页一图 for-loop 跑。Unlimited OCR 以 DeepSeek-OCR 为底，只改一件事——把解码器里所有注意力层换成 R-SWA（Reference Sliding Window Attention）：每个生成 token 只看「全部原图（视觉 token，全局可见） + 刚写的最近 128 个字（滑窗）」，早期输出滑出窗口就淘汰。于是 KV cache 恒定为 L_m + n，不随输出长度涨。配上 DeepEncoder 把一页压成 256 个视觉 token，两头一夹——几十页文档一次前向解析完，还顺手把 OmniDocBench 从 87 涨到 93.23。

§ 01LLM 当解码器的甜头与代价

DeepSeek-OCR 让 OCR 又翻红了：拿 LLM 当解码器，借语言先验，认字更准。但代价同样明显——输出序列越长，累积的 KV cache 越吃显存、生成越来越慢。人抄一整本书不会越抄越慢，模型却会。

于是现有模型没一个能在单次前向里解析十几页，全靠"一页一图、每页把记忆清零"的 for-loop 硬拼。这把一个本该连贯的长程任务，碎成一堆由外部调度器管的孤立短任务——能用，但是工程绕路，不是真的"长程能力"。Unlimited OCR 想把这件事做成模型自己的能力。

§ 02输入侧：一页文字压成 256 个 token

先看输入。反直觉但很硬的事实：一页密密麻麻的文字，当成图编码，比当成文本编码更省 token3。DeepEncoder 把 1024×1024 的 PDF 图压到 256 个视觉 token，约 1:10——一个视觉 token 顶约 10 个文本 token。诀窍是窗口注意力扛海量原图 token、全局注意力只留给压缩后的少量 token，这样编码高分辨率图也不爆显存。

§ 03真正的瓶颈在解码 · 人抄书的工作记忆

问题来了：1:10 意味着 10K 视觉 token（约 20–30 页）要解码出 10 万+ 个文本 token。标准注意力扛不住这么长——KV cache 线性爆炸。怎么办？作者从人抄书里偷招：抄一整本时，注意力只落在三处——一直摊开的源书、刚写下的几个字（确认抄到哪）、下一个要写的字。已经抄完的几十页？软遗忘8，不会每写一字就回头通读。正是"不全记"让人能低负荷连续抄几百页。

§ 04R-SWA：每个字只看「全图 + 最近 128 字」

把这套搬进注意力，就是 R-SWA1。生成第 t 个 token 时，它能看见的集合 = 全部前缀 P2（视觉 token + prompt，永远全局可见）∪ 最近 n 个输出 token6（因果滑窗）。跟标准全注意力比，区别就是右下那张可见性图——全注意力是整个下三角全填，R-SWA 只填"前缀那几列 + 一条宽 n 的对角带"。

§ 05恒定 KV cache：unlimited 的底气

这一步是全部收益的来源。标准 MHA 的 KV cache = L_m + T，随生成长度 T 线性涨；R-SWA 永远只留"前缀 L_m + 最近 n"，即 L_m + n，封顶不动5。算个账：前缀 L_m=300、n=128，解码一本书 T=10000——MHA 要 10300，R-SWA 只要 428，省 96%。T 拉到十万，MHA 跟着涨，R-SWA 还是 428。显存和每步算力都恒定，所以几十页一次过、速度不掉。

为什么不掉点,还涨点

R-SWA vs 普通滑窗注意力的命门：视觉 token 被排除在状态转移之外——它们编码一次后全局静态，不会像普通滑窗那样被反复"揉糊"，所以认字精度保得住。作者还推测：全注意力在输出变长时容易发散，R-SWA 把注意力夹在"原图 + 最近 128 字"里，反而逼模型更专注 dense OCR、靠滑窗里的最近输出定位"我抄到哪了"。所以替换是个免费午餐——省了还涨。

§ 06结果：SOTA + 长程不崩

从 DeepSeek-OCR checkpoint 续训 4000 步、冻结 DeepEncoder 只训 LLM（解码器是 3B-A0.5B 的 MoE7）、2M 文档数据——改动极小却拿了端到端 SOTA。OmniDocBench v1.5 拿 93.2310，比 DeepSeek-OCR baseline 高 6.22、比更强的 DeepSeek-OCR 2 还高 4；v1.6 拿 93.92。更关键的是长程：40+ 页一次性解析，编辑距离仍 < 0.11、Distinct-35 达 97%（没有"复读机"塌缩11）。

放进 wiki 这条线

Unlimited OCR 接的是 光学上下文压缩（DeepEncoder = SAM-ViT 窗口注意力"看清每个字" + CLIP-ViT 全局注意力"理解整页"，细节见概念页）那条线，但只动解码器：把全注意力换成 R-SWA，把 KV cache 从线性压成常数——它是 稀疏注意力 的"前缀全局 + 输出滑窗"两段式特例，跟 FlashAttention 一样在攻注意力的显存/IO 成本，只是一个改可见范围、一个改 IO 调度。一句话：照着人抄书的工作记忆，让每个字只盯「原图 + 最近 128 字」——KV cache 一恒定,几十页就能一口气抄完,而且这刀对 dense OCR 还是免费午餐。R-SWA 不止 OCR,ASR / 翻译这类"盯固定参考、长程输出"的任务都能套。

引用 / 术语表

  1. R-SWA · 参考滑窗注意力 · 每个生成 token 看「全部前缀（视觉+prompt，全局）+ 最近 n 个输出（滑窗，n=128）」，早期输出滑出就淘汰。KV cache 恒定。
  2. 前缀 / 参考 token（P, L_m） · 视觉 token + prompt，长度 L_m。对所有后续 token 全局可见、整个解码过程不变，不参与状态转移所以图不糊。
  3. 光学上下文压缩 · DeepEncoder · SAM-ViT（窗口注意力）串 CLIP-ViT（全局注意力）+ 桥接 16× 压缩，1024×1024 图 → 256 视觉 token，约 1:10 顶文本 token。
  4. KV cache · 解码时缓存的历史 K/V。标准注意力随输出长度线性涨（越跑越慢、越吃显存），是长程 OCR 的瓶颈。
  5. 恒定 KV cache · R-SWA 下 cache = L_m + min(n,T) ≤ L_m + n，封顶不随 T 涨。L_m=300/n=128/T=10000 时 428 vs MHA 10300，省 96%。
  6. 滑窗宽度 n · 输出区因果窗口大小，默认 128（多页推理用 1024）。单页 OCR 全用 n=128 替换全注意力，无损还涨点。
  7. MoE 3B-A0.5B · 继承 DeepSeek-OCR 的 混合专家解码器，3B 总参、推理只激活 0.5B，效率高。
  8. 软遗忘 / 工作记忆 · 人抄书只盯源书+刚写的几个字+下一个字，已抄完的不通读。R-SWA 把它落成"前缀全局 + 输出滑窗 + 淘汰旧输出"。
  9. DeepSeek-OCR · 本文的 baseline（2025）：DeepEncoder 光学压缩 + 3B-MoE 解码器，让端到端 OCR 翻红；但 KV cache 随输出线性涨。
  10. OmniDocBench · 文档解析基准，多维（文本编辑距离、公式 CDM、表格 TEDS、阅读顺序）。Unlimited v1.5 拿 93.23、v1.6 拿 93.92 端到端 SOTA。
  11. Distinct-n · 生成文本里不重复 n-gram 的占比，衡量长程解析有没有"复读机"塌缩。40+ 页时 Distinct-35 仍 97%。
