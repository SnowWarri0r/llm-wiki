Cosmos 3 · Omnimodal World Models for Physical AI · 139-page Technical Report

真正的统一，不是把五种输入都塞进一个框。它得让“推理下一句话”和“生成下一段世界”在同一个模型里各干各的，又能互相通气。

TL;DR

Cosmos 3 给物理世界里的 AI做底座：同一个 checkpoint 里放一座负责理解的 Reasoner 和一座负责生成的 Generator。语言走自回归，图像、视频、音频和动作走去噪生成。两座塔参数分开，注意力却能交换信息。它统一的是模型架构、token 接口和训练起点；并没有把多步生成变成一次前向，也没有让一个未经专项训练的 checkpoint 包揽所有榜单。

§ 01机器人收桌子，为什么要串三套脑子

假设机器人接到一句话：“把杯子放进托盘。”它先要看懂桌面，认出杯子和托盘；再想象机械臂往哪走才不会碰倒水壶；最后输出一串动作。旧做法往往把这三步交给不同模型：VLM 看懂，世界模型画未来，策略模型出动作。

问题不只在部署麻烦。三个模型用三套内部表示，信息每过一道接口就会缩水。VLM 说“杯子在左边”，世界模型却不知道“左边”对应哪个像素和哪条机械臂轨迹；策略模型又未必知道它的动作会把画面改成什么样。

先把“统一”说准Cosmos 3 不是把所有能力揉成一套权重。恰恰相反，它用两套完整参数隔开理解与生成，再让两边通过联合注意力交换信息。

§ 02先看全貌：五个入口，两座塔，很多出口

整条链可以压成三步。第一步，各模态用自己的编码器变成token 或 latent token；第二步，把条件和目标排进同一条序列；第三步，Reasoner 或 Generator 处理自己负责的 token。这个“双塔”结构就是Mixture-of-Transformers。

§ 03五种模态，先各自翻译成 token

“五模态统一”最容易写成口号。真正落到模型里，每种数据的入口并不一样。图片拿来理解和拿来生成，甚至走的是两条不同的路。

动作为什么不能共用一个固定 tokenizer

不同机器人的控制面板根本不一样。单臂机器人一步动作是 10 个数：3 个平移、6 个旋转表示、1 个夹爪状态；双臂是 20 维；人形机器人是 29 维；第一视角人体动作最多 57 维。Cosmos 3 给每类设备配一对输入/输出投影：进模型前统一到相同隐藏宽度，出模型后再还原成原设备需要的维数。

为什么这么做统一的是 Transformer 中间的“语言”，不是强迫所有机器人使用同一种控制协议。这样换设备时主要替换入口、出口和后训练数据，不必重造主干。

§ 04同一条序列，换个摆法就换任务

Cosmos 3 先放 AR 子序列，再放扩散子序列。扩散部分内部又遵守一条规矩：干净条件在前，带噪目标在后。任务的差别，本质上是哪些 token 已知、哪些 token 要去噪。

文生图和图生视频也没有另起炉灶：文生图是“文字 + 噪声图”；图生视频是“文字 + 干净首帧 + 噪声后续帧”。这才是 token 排列真正改进的地方——不是凭空创造新能力，而是把很多任务改写成同一个训练接口。

§ 05MoT：为什么非得分成两座塔

理解和生成对 Transformer 的要求相反。写下一句话时，当前位置不能偷看未来；去噪一张图时，左上角却应该知道右下角有什么，不然整张图很难协调。MoT 的做法很干脆：每层的归一化、注意力投影和 MLP 都准备两套，AR token 和扩散 token 各走一套。

把账算清：64B 总参数，不代表每个 token 都跑 64B

Reasoner ≈ 32B　+　Generator ≈ 32B　→　总参数 ≈ 64B 100 个 AR token　→ 只过 32B Reasoner 400 个扩散 token → 只过 32B Generator 所以：每个 token 激活的主干容量约为 32B，不是 64B。 但是：部署仍要保存两塔权重，权重显存不会凭空减半。

✓ 容量翻倍和每 token 计算不翻倍可以同时成立；显存代价仍在。

初始化也要说准Nano/Super 的 Reasoner 分别从 Qwen3-VL-8B/32B 出发，Edge 使用 NVIDIA 自己预训练的模型。先把 Reasoner 训练好，再把它的权重复制给 Generator 做初始化。不是“两塔同时从 Qwen3-VL 开始训练”。

§ 06联合注意力：两边能交流，但不是谁都能看谁

两塔参数分开后，还需要一张“会议桌”。Cosmos 3 的联合注意力把两边产生的 key/value 拼起来，但对 query 设不同权限。这就是因果注意力和双向注意力同时存在的办法。

最关键的不对称是：Generator 能读 Reasoner，Reasoner 不会被尚未去噪完成的 Generator 反向更新。这样生成端拿得到语义，语言端又不会被一团噪声带偏。

§ 08帧编号不够，模型得知道真实过去了多久

两个视频都有 5 个 latent 帧，不代表它们经历了相同时间。16 FPS 的相邻帧比 30 FPS 隔得更久。Cosmos 3 用三维MRoPE编码时间、高度、宽度，再按真实FPS / token 速率缩放时间轴。

§ 09数据：不只要多，还得知道哪里该严

Reasoner 的预训练集共有 22,002,013 个样本，SFT 有 2,171,673 个。论文没有把数据清洗写成一句“高质量筛选”，而是把去重、打分和阈值都交代了。

为什么预训练放宽，SFT 反而更严

AI judge 给忠实、完整、正确三项各打 1–5 分。预训练阈值设为 2，保留约 78%：这时模型需要广覆盖，不能为了“全是满分样本”把 OCR、空间定位等长尾能力砍没。监督微调阈值设为 5，只保留约 46%：SFT 会直接塑造模型怎么回答，错一条监督比少一条更糟。

Generator 还要吃图像、视频、音频、动作和控制视频。视频覆盖 256p、480p、720p；最长分别为 400、400、300 帧。不同分辨率和长度的样本被拼进固定 74,000-token 上下文，尽量少把算力浪费在 padding 上。

Generator 的数据量，先看“漏斗”而不是最后一个数字

生成数据不是从网上抓下来就直接开训。图像从 78 亿张原始素材里留下 7.67 亿张；视频从 30 亿段来源素材里留下 3.477 亿段；音频预训练集则有 1.389 亿段可用片段。到了 mid-training，数量进一步收窄，换来更细的领域标注、更高的画面质量和更明确的动作语义。

为什么把 caption 写成 JSON，而不是一段漂亮散文

自由文本 caption 很容易漏掉位置、动作先后和声音来源。Cosmos 3 用两套微调过的 Qwen3-VL-8B 生成结构化 caption：场景、主体、动作、镜头、时间变化、声音等各写进固定字段。检查时也不只问“读起来像不像”，而是把描述拆成原子事实，分别核对精确率和召回率。

不同模态不能照抄同一套清洗规则

图像 / 视频

先用 TransNetV2 切镜头、去黑边并统一编码；再用视觉 embedding 聚类去重。视频额外用 DOVER 评审美和技术质量，用 VTSS 给运动质量打 0–9 分，还标注约 100 类伪影。

音频

先做声源分离，再检查嘴型同步、语音 / 音乐成分与乐器类别，最后对处理后的波形重新写 caption。原因很直接：处理前的描述未必仍对应处理后的声音。

动作

8.4M 条 episode、61.3K 小时，覆盖第一视角、自动驾驶、机器人和相机运动。不同设备动作先归一化到大致 [-1,1]，但保留静止片段的元数据，避免把“停住”误删成无效数据。

合成数据

报告还放出 PhyxSim、RobotSim、DriveSim、SynHuman、Warehouse 五类模拟数据。它们能补稀有事件和精确标签，但不能单独替代真实数据：人类动作仍有明显 sim-to-real 差距。

§ 10训练不是一起开跑，而是一棒接一棒

“两塔统一训练”很容易让人误以为模型从头到尾一次训完。实际流程更像接力：先把理解练扎实，再把这套世界知识交给生成塔，最后才加入动作和专项后训练。

每一棒具体喂什么、更新谁

报告里最容易被“预训练—中训—后训练”三个词遮住的，是每一阶段其实换了数据配比、噪声分布、分辨率和可训练参数。下面把它们摊开。

74K token 打包到底解决什么：拿三条样本走一遍

视频样本长度能相差上百倍。若每张卡只放一条样本，短视频后面全是 padding；若每张卡随便拼，所有 GPU 又可能拿到不同形状，集体通信无法对齐。Cosmos 3 让各 rank 同步选择数据流，并用前看式打包在固定 token 预算内找下一个能塞下的样本。

QKV 复制过去，怎么从“预测下个词”变成“预测去噪速度”

它不会在复制的一瞬间变成去噪器。复制只是给 Generator 一个比随机数更懂语义的起点；后面的图像、视频、音频生成预训练，才真正改造这些参数。

QKV 参数本身并没有写着“只能预测下一个词”。Transformer block 一直只是在重组隐藏向量、提取关系。任务由输入、attention mask、输出头和损失共同决定。视频与音频 latent 通过线性层进入同一个隐藏维度；动作的输入、输出投影则由随机初始化开始联合训练。复制 Reasoner 的价值，是不用从零重学“机械臂、杯子、托盘分别是什么，以及它们怎样关联”；但去噪方向仍要靠 Flow 训练做迁移初始化后的再学习。

论文证据的边界主训练流程明确写的是“用训练好的 Reasoner 权重初始化 Generator”。但附录 E.1 并没有直接比较“随机初始化 Generator vs 复制 Reasoner 初始化”：那组实验让两个 Generator 都从头训练，只替换条件侧的理解塔。它证明更好的 Reasoner 表征能提高 Physical AI domain 分数，不能单独证明复制初始化贡献了多少。
最容易夸大的地方“一个模型支持很多任务”是真的；“同一个未经专项训练的 checkpoint 同时拿下所有第一”不是真的。论文表格里的星号明确标出后训练版本。

§ 11模型能训起来，靠的不是一句“分布式训练”

139 页报告里相当多篇幅在讲系统。原因是它同时处理文字、图片和数百帧视频，单条样本长度能差两个数量级；两塔注意力还有不同 mask。若数据管线、attention kernel 和 checkpoint 任一环节跟不上，算法配方根本跑不到规模。

SILA：把数十亿条素材做成可追踪的生产线

SILA 是 Scalable Infrastructure for Large-scale data processing and Annotation。它把原始媒体、embedding、caption、质量分和处理状态放进同一张有类型的 Lance 宽表，不再为每条管线维护一张 PostgreSQL 表再反复 join。

两种注意力规则，为什么要拆成两个 kernel

Reasoner 要因果 mask，Generator 要双向 mask。如果把两者塞进一个通用 FlexAttention mask，灵活但慢。系统把序列重排成 [R₀,G₀,R₁,G₁,…]，分别调用两次变长 SDPA kernel，再把结果放回原位：Nano 上比 FlexAttention 快约 22%。Hopper 用 FlashAttention-3，Blackwell 则用 NATTEN。

分片策略

参数与优化器状态用HSDP切分，长序列用 Ulysses context parallel 切 token；报告没有采用 ring attention，因为这里的交错序列和不对称 mask 会增加通信与实现复杂度。

少存一点中间结果

Selective Activation Checkpointing 只保留注意力输出，报告称比完整重算多约 13% 开销；若整层 checkpoint，额外 FLOPs 约 33%。

编译

torch.compile 让训练吞吐提高约 41%。视频 tokenizer 按分辨率分块后，编译使编码延迟降低 52%；45 种图形用分布式 AOT 缓存，把启动从约 15 分钟压到 1 分钟内。

异步 checkpoint

写盘与下一轮训练重叠。Nano 平均约 72 秒、Super 约 167 秒完成一次保存；对端到端训练的额外开销分别约 4% 和 9%。

报告内部有一处数字冲突训练章节写 Nano / Super 分别使用 1024 / 2048 张 GPU；吞吐表的 caption 却写 2048 / 4096 张。正文没有解释是否换了测量规模，因此这里只并列报告，不替作者擅自选一个“正确值”。

§ 12部署时，延迟和吞吐不是一回事

生成 189 帧视频时，一个请求要占用大量 token。延迟问单个用户多久拿到结果；吞吐问整组 GPU 每小时能处理多少请求。batch 变大通常提高吞吐，却不保证某一个请求更快。

两套服务路线，各自解决什么

路线做了什么适用边界
PYTORCH REFERENCE逐 block 编译 / CUDA Graph、context parallel、CFG 双卡并行、Reasoner 条件缓存、变长 batch。代码最接近研究实现，便于改模型和查错；不是为了把所有场景都压到最低延迟。
vLLM / TRT / OMNIReasoner 用 vLLM 或 TensorRT；Generator 用 vLLM-Omni，加入 Cache-DiT、Ulysses、HSDP、CPU offload、VAE patch parallel 与 FP8。更适合稳定服务与多卡部署，但量化、缓存和并行策略都要按任务验证质量与收益。

批处理收益也不是固定常数。189 帧、256p 文生视频时，报告看到 Nano 在 H100 上约 +8%、GB200 上约 +40%，Super 在 H100 上约 +55%、GB200 上约 +9%；到了 480p 只剩约 1–5%，720p 的 batch=1 则没有批处理收益。瓶颈会随分辨率、模型大小和硬件改变。

§ 13六类实验，分别回答六个问题

Cosmos 3 横跨理解、图像、视频、音频、视频控制和动作。把不同 benchmark 混成一个“总冠军”没有意义；更合理的读法是先问每张表测的是什么，再看优势是否稳定。

3 · 音频：语义和对齐强，制作质感还不是最好

SoundBench 用 144 条提示，先分别评声音语义、音画对齐、视觉支持和制作质量，再组合成：

SAV = 0.6×声音语义 + 0.3×音画对齐 + 0.1×视觉支持
AVQ = 0.5×SAV + 0.5×制作质量

Nano 的 SAV 为 8.35、制作质量 6.32，最终 AVQ 7.34；Seedance 1.5 Pro 的 AVQ 为 7.64、制作质量 7.06。也就是说 Cosmos 3 更懂“该响什么、何时响”，但底噪、空间感与声音质感仍拖后腿。

4 · Transfer：控制更准，代价是一次 CFG 要跑三次 denoiser

普通 CFG 只有“有条件 / 无条件”两路。视频 transfer 既有文字条件又有边缘、深度、分割等控制，因此使用双权重 CFG：无条件、仅文本、文本+控制各跑一次，再分别调文字强度和控制强度。好处是两种约束可以分别调节；代价是每个去噪步要三次前向。

代表结果PAIBench-C 上 Super 的分割 / 模糊控制约为 0.71 / 0.91，Nano 为 0.72 / 0.91；AVBench 的 ego motion drift 都是 0.003，低于对照的 0.008。不同控制指标量纲不同，不能横向比较“0.91 一定比 0.71 好”。

5 · Action：不仅会预测动作，还要证明能换设备

6 · 这些分数有哪些共同边界

PAIBench overall 是 0.5×Quality + 0.5×Domain，Domain 依赖 VLM judge。报告还说明公开榜单的一部分 I2V 分数无法复现，因此内部统一用 Qwen2.5-VL-72B 重算。Physics-IQ 的 Best-of-N 叠加了额外生成与重排；动作 benchmark 的离线误差也不等于真实闭环成功率。分数可以回答“在这套协议下谁更好”，不能直接证明现实世界已经可靠。

§ 14实际怎么用：先选任务，再选采样配方

模型支持不等于参数随便设。报告给出的有效范围是 10–30 FPS、5–400 帧、256p / 480p / 720p 和五种宽高比。分辨率、帧数、去噪步数和 CFG 都会同时影响显存、速度与画面稳定性。

提示词为什么也要结构化

报告建议用 Reasoner 或 Claude 把短提示扩写成 JSON：先写场景与构图，再写按时间展开的动作计划，最后补声音。比如“机器人把杯子放进托盘”至少要展开成：主体外观与初始位置、机械臂从哪侧接近、何时夹紧、杯子怎样移动、何时松开、镜头是否跟随、接触时出现什么声音。这样做不是让文字更华丽，而是把模型容易漏掉的时空约束显式列出来。

实用边界长提示并非一定更好。互相冲突的镜头、动作和声音要求会降低可控性；先保留必须满足的约束，再补风格细节。机器人策略更不该依赖文学化描述，而要用训练时见过的任务语言和相机布局。

§ 15消融最诚实：统一有收益，也有账单

消融实验不是看谁赢，而是只改一个零件，确认提升到底从哪来。Cosmos 3 最值得看的五组如下。

最后把边界摊开

权重显存没有省每 token 只走一塔，但部署仍要装下两塔。报告没有给单流统一模型的严格端到端延迟对照。
生成仍是多步图像、视频、音频和动作都要迭代去噪。“统一模型”不等于“一次 forward 完成所有输出”。
冠军来自专项后训练文生图、图生视频和机器人策略的最佳结果使用 post-trained 变体，不能全算到通用 checkpoint 头上。
评测仍依赖 judge多项视频分数来自 VLM 或人工协议；复现、评审模型和 Best-of-N 都会改变结果。
合成数据不是免费午餐总体平均略升，但人类动作指标下降；模拟器覆盖与真实分布仍有缝隙。
离可靠机器人还远RoboLab specific 指令成功率 39.7% 已领先，但也意味着多数复杂 rollout 仍会失败。
一句话带走Cosmos 3 的核心不是“五模态第一”，而是把理解、生成和动作写进同一套 token 与训练接口，再用两座参数隔离、注意力相通的 Transformer 承担不同工作。真正决定它能不能落地的，是数据漏斗、阶段配比、长序列系统和专项后训练一起发挥作用。

术语表 · 点正文角标可随时查看

  1. Physical AI · 物理 AI · 在真实世界感知、推理和行动的系统，例如机器人、自驾和智能空间。动作会改变环境，错误也可能造成真实后果。
  2. World model · 世界模型 · 给定当前状态和动作，预测接下来会发生什么。它可以生成未来画面，也可以用来评估动作后果。
  3. Mixture-of-Transformers · MoT · 为不同 token 类型准备完整的 Transformer 参数路径。Cosmos 3 里是 Reasoner 和 Generator 两塔，不是 MoE 的 FFN 专家路由。
  4. Autoregressive · 自回归 · 一步接一步预测下一个 token。当前位置只能看前文，适合语言生成。
  5. Rectified flow / 去噪生成 · 在数据与噪声之间连直线路径，让模型学习沿途速度；推理时从噪声出发，多步走回数据。
  6. Token / latent token · Transformer 处理的基本单元。文本 token 是离散编号；视频、音频等通常先压缩成连续 latent，再当作 token。
  7. ViT · 把图片切成小块，再用 Transformer 理解这些块之间的关系。Cosmos 3 用它做视觉理解入口。
  8. VAE · 把像素或波形压进更小的连续 latent 空间，也能再解码回来。Generator 在 latent 空间去噪，比直接生成像素省得多。
  9. Action token · 动作 token · 平移、旋转、夹爪等连续控制量先投影到 Transformer 隐藏空间；生成后再投影回设备动作维数。
  10. Joint attention · 联合注意力 · 两塔各算自己的 query/key/value，再按权限让 Generator 读取 Reasoner 的条件信息。
  11. 因果 / 双向注意力 · 因果注意力不能看未来；双向注意力能同时看整块 token。语言用前者，扩散生成用后者。
  12. 3D MRoPE · 把旋转位置编码拆到时间、高度、宽度三个轴。语言令三轴位置相同，视频使用真实的三维位置。
  13. FPS / TPS · FPS 是每秒原始帧数；TPS 是视频经过 VAE 时间压缩后，每秒剩多少 latent token。
  14. SFT · 监督微调 · 用高质量“输入－标准答案”继续训练，让基础模型学会目标任务和回答方式。
  15. Mid-training / post-training · Mid-training 在大规模预训练后加入新模态和新能力；post-training 用更窄、更精的数据做专项强化。
  16. Ablation · 消融实验 · 只拿掉或替换一个组件，其余条件尽量不变，用来判断提升到底是不是这个组件带来的。
  17. 参数矩阵 vs attention 图 · WQ/WK/WV/WO 是会保存、复制和更新的模型参数；softmax(QKᵀ) 是每次前向按输入与 mask 临时算出的注意力分布。
  18. 迁移初始化 · 复制旧任务学到的参数当新任务起点，不等于已经掌握新任务；新的输入、输出头与损失会继续改造全部主干参数。
  19. 结构化 caption · 不把整段描述写成自由散文，而是把主体、空间、动作顺序、镜头和声音放进固定字段。它更容易逐项检查，也更适合稳定地转成训练条件。
  20. Look-ahead packing · 前看式打包 · 当前样本塞不进剩余 token 容量时，先暂存它，再向后找较短样本填空；暂存样本留给下一包。目的是少 padding，同时不丢样本。
  21. SILA / Lance · SILA 是 Cosmos 的大规模数据处理与标注系统；Lance 是适合媒体、向量和表格数据的列式存储格式。这里用于把素材、embedding、标注和处理状态放在同一份可追踪数据里。
  22. HSDP / Context Parallel · HSDP 把模型参数、梯度和优化器状态分散到多张卡；context parallel 把同一条超长序列的 token 分开计算。一个解决“模型放不下”，另一个解决“序列放不下”。
  23. CFG · Classifier-Free Guidance · 分别预测“有条件”和“无条件”的去噪方向，再放大两者差值，让结果更服从提示。guidance 太大可能使画面生硬或失真。
  24. Cosmos-HUE · 把一条视频提示拆成可直接回答 Yes / No / Unclear 的原子问题，再由两名标注员和质检复核。它比“整体打几分”更容易知道模型究竟错在物体、动作还是时序。
  25. Two-weight CFG · 双权重 CFG · 对无条件、文字条件、文字+控制条件做三次预测，再分别调节文字服从度和控制强度。多一个旋钮，也多一次 denoiser 前向。
