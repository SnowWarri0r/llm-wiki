ViiTorVoice ·
低帧率语义码,并行填声。

viitor-ai · An LLM-based TTS engine (viitor-voice-nar) · 思路源自 DualCodec + OmniVoice · HF: ZzWater/ViiTorVoice-NAR

实时可克隆的 TTS,卡在两条瓶颈:帧率高→序列长、逐帧自回归→生成慢。ViiTorVoice 把三样拼一条链:DualCodec 把音频压到 12.5 帧/秒的"语义码+声学码"、NAR 掩码并行一次性填声、首块流式吐音——端到端首帧约 60ms,给段参考音就克隆。

TL;DR

一个开源流式零样本 TTS 引擎(没有独立论文,硬数字来自它搭积木用的两篇组件论文)。三招各砍一刀延迟:① DualCodec 低帧率码——24kHz 压 1920 倍到 12.5 帧/秒(比 Encodec ~75Hz 短 6×),第 1 码本是语义(w2v-BERT-2.0 蒸馏,码本 16384)、2–8 码本补声学残差24;② NAR 掩码并行——不逐帧,像完形填空一次性并行填码、迭代几步收敛,步数从 200 降到 ~817;③ 首块流式——首帧延迟 ~60ms8。给参考音零样本克隆9,两路 CFG 调情绪/副语言10。

§ 01痛点：帧率高 + 逐帧生成,实时克隆两头卡

LLM-based TTS 的常见做法:先把音频用神经码(codec)压成离散 token,再让模型逐帧自回归吐出来。两个瓶颈叠在一起1:

  • 帧率高 → 序列长:Encodec 这类声学码在 24kHz 下约 75 帧/秒,一句几秒的话就是几百帧,序列长、模型难学、显存吃紧。
  • 逐帧自回归 → 生成慢:第 n 帧要等第 n−1 帧算完,200 帧就是 200 步,长音频天然慢、首帧延迟高。

要做"实时、可克隆、可控情绪"的语音,得同时砍掉"序列太长"和"逐帧太慢"。ViiTorVoice 的办法是三招各砍一刀:低帧率语义码(§02)、NAR 并行填声(§03)、首块流式(§04)。

§ 02招一：DualCodec 低帧率码,把"内容"和"细节"分两条流

DualCodec2 把 24kHz 波形压 1920 倍到 12.5 帧/秒——比 Encodec ~75Hz 短 6×,下游序列直接短一个数量级3。更关键的是它双流编码:第 1 码本(RVQ-1)不是普通声学码,是从自监督模型 w2v-BERT-2.0 第 16 层蒸馏来的语义码(码本 16384),管"说什么"4;第 2–8 码本量化"波形减掉语义"剩下的声学残差(音色/韵律/环境)5。一共 8 个码本,走 RVQ 残差递推6。

数字例子 · 12.5 帧/秒的账

输入 24000 样点/秒
编码器总下采样 = 4×5×6×8×2 = 1920
帧率 = 24000 / 1920 = 12.5 帧/秒     （25Hz 版去掉最后 ×2：24000/960 = 25）
对比 Encodec 24kHz：下采样 320 → 75 帧/秒
一句 2 秒话：DualCodec 2×12.5 = 25 帧（每帧 8 码）; Encodec 2×75 = 150 帧
省 75/12.5 = 6× 帧数 → LLM 面对的序列短一个数量级

✓ 帧率是"每秒吐几帧码"。压得越狠(下采样 1920)帧率越低、序列越短。语义码本(RVQ-1)从 w2v-BERT 蒸馏这招,让下游只需先把很短的语义序列预测对,内容就对了——声学细节留给 §03 并行补。

§ 03招二：NAR 掩码并行填声,不逐帧

声学码那 175 个码,若逐帧自回归就是 175 步。ViiTorVoice 走非自回归(NAR)1:先把要生成的码全盖住(掩码),一次性并行预测全部,再把没把握的盖回去重猜、迭代几步收敛——像 BERT 完形填空,但用于生成7。步数从"序列长度"降到"迭代轮数(~8)",且每步内部并行。这一脉是 SoundStorm / MaskGCT / OmniVoice12,从预训练 LLM 初始化。

§ 04招三：首块流式 60ms + 零样本克隆 + CFG 控制

首块流式:不必等整段生成完再出声,算完第一块就开始吐音、后面边算边流——端到端首帧延迟 ~60ms8。零样本克隆:给一段参考音频(或预算好的 prompt 码本),模型照着音色合成目标文本,不用微调9——因为克隆改的是声学侧,内容(语义码)不动11。可控:两路 CFG guidance(情绪 emotion_guidance_scale、非语言发声 nvv_guidance_scale)调情绪/副语言强度10;局部编辑靠对齐后掩码(词级 align_granularity、expand_mask_ratio)。

§ 05结果与落地

把低帧率语义码 + NAR 并行 + 首块流式拼起来,得到一个实时、可克隆、可控的开源 TTS 引擎,拆成 gRPC v2 服务(encoder / llm / decoder / orchestrator)+ HTTP 网关(:7861)。硬数字大多来自它搭积木用的两篇组件论文:

ViiTorVoice 没有独立论文,亮点是工程整合:把 DualCodec(低帧率 + 语义/声学分流)+ NAR 掩码并行(OmniVoice/SoundStorm 一脉)+ 首块流式,拼成一条 60ms 的实时克隆链。跟 fish-speech 对照着看:都用 RVQ 码 + 参考音克隆,但 ViiTorVoice 把生成从自回归换成 NAR 并行、把帧率压到 12.5Hz 抢延迟。一句话:内容用短语义序列先定、细节用 NAR 并行补、首块先发声——三样各砍一刀,凑出实时。

引用 / 术语表

  1. 自回归 vs 非自回归(AR/NAR) · AR 逐帧生成,成本 ∝ 序列长度;NAR 并行预测全部、迭代几步收敛,成本 ∝ 迭代轮数(常数)。长音频/低延迟场景 NAR 赢。
  2. DualCodec · 低帧率、语义增强的双流神经音频码(arXiv 2505.13000 · Interspeech 2025)。SSL 流 + 波形流,RVQ-1 语义、其余声学。
  3. 低帧率 12.5Hz · 24kHz 下采样 1920(strides 4·5·6·8·2)→ 12.5 帧/秒,比 Encodec ~75Hz 短 6×,下游序列短一个数量级。
  4. 语义码(RVQ-1) · 第 1 码本从自监督模型 w2v-BERT-2.0 第 16 层蒸馏,码本 16384,编码"说什么"(内容骨架)。
  5. 声学码(RVQ 2-8) · 量化"波形 − 语义"的残差,补音色/韵律/环境等细节。共 8 码本。
  6. RVQ 残差量化 · 一层层量化上一层的残差;码本越多还原越细。DualCodec 底层就是它。
  7. 掩码并行生成 · 全盖住→并行猜→没把握的重盖再填,迭代几步(~8)收敛。SoundStorm/MaskGCT/OmniVoice 一脉,像 BERT 完形填空但用于生成。
  8. 首块流式 · 60ms · 算完第一块就发声、后面边算边流,端到端首帧延迟约 60ms,不等整段。
  9. 零样本克隆 · 给一段参考音频(或 prompt 码本)照着音色合成目标文本,不用微调。
  10. 两路 CFG 控制 · emotion_guidance_scale(情绪)+ nvv_guidance_scale(非语言发声),用无分类器引导调控制强度。
  11. 局部编辑(掩码对齐) · 把源音按词级对齐(align_granularity)后掩掉要改的段(expand_mask_ratio),只重填那部分,内容其余不动。
  12. OmniVoice · 扩散语言模型式的离散 NAR 架构(arXiv 2604.00688 · k2-fsa),从预训练 LLM 初始化、全码本随机掩码训练;ViiTorVoice 的 NAR 思路来源之一。
