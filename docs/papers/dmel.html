Speech Tokenization
Made Simple.

Bai · Zhao · Bao · He · ... · Apple · 2024-07

把语音切成离散 token 通常要训个 neural codec（fish-speech / EnCodec / SoundStream）。dMel 提出：直接把 log-mel 按预设 bin 量化成 token，不用训 codec。然后证明这套简单方案在 speech-LLM 训练里跟 RVQ codec一样好。

TL;DR

2024 主流 "speech as LLM input" 方案是 训个 neural codec 把音频压成离散 token（RVQ / EnCodec / SoundStream）。又贵又复杂，跟 LLM 训练分离。

dMel 的方案：跳过 codec，直接 bin quantize log-mel 频谱。每个 mel 频道按 16 个等距 bin 切，token 就是 bin index。20 行 numpy 代码就能复现。

结果：在 ASR / TTS / speech LLM 训练上，dMel 跟 codec 路线性能相当。TML 用 dMel 做音频输入 + flow matching 做音频输出 —— 全链路无独立 codec。

这是个反复出现的范式：主模型变强后，预处理的复杂度可以塌缩。

§ 01音频 → token · 主流做法的痛点

2024 主流 speech-LLM 训练管线：

  1. 训一个 neural codec（RVQ / EnCodec / SoundStream / fish-speech codec）把音频压成离散 token
  2. 用这些 token 训 speech LLM

问题 ——

  • codec 是独立模型，要训练、调参、维护，自己就是个大工程
  • codec 训练和 LLM 训练分离，不能端到端反向传播
  • codec 决定上限 · LLM 见到的"音频"是 codec 抽象出来的样子，可能不是 LLM 视角下最优的表征
  • 跨语言迁移困难 · 英语训的 codec 跑中文可能掉点（codebook 跟训练域绑定）
  • 可解释性差 · codebook entry 是个学出来的黑盒向量，debug 时不知道每个 token 对应啥

Apple 内部的疑问：能不能不用 codec？ dMel 给了答案。

§ 02从波形到 log-mel · 输入是什么

音频原始是 1D 波形（16kHz 采样率 = 每秒 16000 个浮点）。太长。dMel 接受的输入是 log-mel 频谱图1 —— 一个 2D 矩阵 `[频道 × 时间]`。

到这里得到的还是连续浮点矩阵。要喂 LLM（吃离散 token），必须离散化。这一步就是 dMel。

实际配置：80 mel channels × 16 bins × N frames。每帧 80 个 token（每个 channel 一个），vocabulary 大小是 167。10 秒音频 ≈ 80 × 1000 = 80000 token。vmin / vmax 不是学的，是数据集统计值8（取 1%/99% 分位数避开极端）。

关键属性
  • 不需要训练 · vmin/vmax 是数据集统计值
  • 跨语言稳定 · log-mel 是物理量，bin 切片纯几何，跟语言无关
  • 可解释 · token = bin index = 这个频道在那帧的能量等级，直接对应频谱图
  • 跨域稳定 · 英语训的模型在中文 dMel 上直接 work

§ 04dMel vs RVQ · 复杂度差一个量级

为什么 dMel 能跟 RVQ 打平

两个原因：

1 · 主 LLM 足够强。当 transformer 大到一定程度，它能从 raw bin index 序列里学到复杂结构 —— 不需要 codec 帮它"压缩信息"。codec 的"额外表达力"在大 LLM 视角下并不必要。

2 · 信息没丢。16 bin × 80 channels 足够表达 mel 频谱的有用变化。RVQ codec 的"额外质量"主要体现在生成端的 vocoder 解码，对 LLM 训练阶段不重要。

§ 05这是个反复出现的范式

"主模型变强后预处理可以塌缩"这事在 ML 历史上反复出现：

每一列都是"原本以为很重要的预处理，被主模型变强吃掉了"。dMel 在音频 token 化这一具体位置上重新讲了一遍这个故事4。

设计原则

遇到"我应该训个 codec / 设计个预处理 / 写规则" 的冲动时，先问：主模型足够强吗？强到不需要这个辅助？

如果是，直接最 naive 的方案试一遍。dMel 的全部贡献就是这个发现："直接 bin quantize 试一下" → 它 work。Apple 的工程师没发明新东西，只是抗住了"得训个 codec 才行"的直觉，跑了 ablation 证明给大家看 —— 这种 "baseline-first" 工程哲学12在 paper 写作上不性感，工程上极有价值。

§ 06TML 怎么把 dMel 接进 micro-turn

TML interaction model 是 dMel 最有意思的应用之一。TML 的音频管线设计：

音频输入 →  log-mel 频谱  →  dMel bin quantize  →  token 流  →  micro-turn transformer
                                                          (并发跟文本 token 在同一序列里)

模型推理 →  生成连续 mel 特征  →  flow matching ODE 积分  →  vocoder9  →  音频输出
                                  (4-8 步 · 200ms 预算内)

整条链路没有任何独立 codec 模型：

  • 输入侧 · dMel 让"音频 → token"是规则查表
  • 主模型 · transformer 直接吃 dMel token 跟文本 token，端到端联训
  • 输出侧 · flow matching 在连续空间生成 mel，可微

这个组合让 TML 实现真正的端到端语音 LLM5 —— 没有梯度断点，没有独立模型，没有 codec 训练。dMel 是这个 elegant 设计的关键支柱。

dMel 不是终极方案

需要承认 trade-off：

  • RVQ codec 在音频生成质量上仍领先（codebook 学了更精细的 phonetic / prosodic 模式）
  • dMel 在简单度 + 实证可用上领先
  • fish-speech 走 RVQ + dual-AR 是另一条同样合理的路线

选哪条看你最在乎什么 —— 工程简洁 vs 生成 SOTA11。TML 选了 dMel 因为200ms 实时预算下端到端可微更重要，fish-speech 选 RVQ 因为生成质量是产品核心。

§ 07历史定位与影响

  • 2017 Tacotron · Mel spec + WaveNet vocoder · 经典 TTS pipeline
  • 2020 EnCodec / SoundStream · neural codec · RVQ-based · 高质量音频压缩
  • 2022 AudioLM · 用 SoundStream token 做音频 LLM
  • 2023 fish-speech / Moshi / Spirit-LM · 各自 codec + LLM 组合
  • 2024-07 dMel · 简单 bin quantization 跟 codec 性能相当
  • 2024-12 TML · 用 dMel + flow matching 做实时语音 LLM6
  • 2025+ · 简单 + 端到端方案逐步成为研究主流
真正的遗产

dMel 当时学术界没特别炸 —— 因为它"不像 contribution"（就是简化了一下）。但工业界很快接受了它，因为简化意味着部署省事 / 调试可控 / 跨域可用。

它的影响不在"打榜 SOTA"，在给社区一个 strong baseline 让人停下来想"我真的需要 codec 吗？" —— 类似 RoBERTa 给 BERT 时代的影响（"先把简单方案做扎实再考虑复杂"）。这种pipeline 简化的工程价值10累计起来比 paper SOTA 重要得多。

引用 / 术语表

  1. Log-mel spectrogram · 对数 Mel 频谱图 · 把波形按短时窗 FFT → 转到 Mel scale（人耳对数感知）→ 取 log 后得到 `[80 频道 × N 帧]` 的连续矩阵。10 秒音频 ≈ 80 × 1000 = 80000 浮点。所有现代 speech LLM 的标准输入特征。详见 log-mel-spectrogram。
  2. RVQ codec · Residual Vector Quantization · neural codec 主流方案。用 encoder 把音频编码成向量序列，每个向量在多层 codebook 上做残差量化。EnCodec / SoundStream / fish-speech codec 都用 RVQ。表达力强但要训练 + 跟训练域绑定。详见 rvq-codec。
  3. "性能相当"的实证 · dMel paper 在 ASR (Librispeech) / TTS (LibriTTS) / speech LLM (内部 benchmark) 上跟 EnCodec / SoundStream / 自家 RVQ 对照，结果是多数指标差距 ≤ 2%。这是个 "good enough" 的对照 —— 不是 dMel 全面胜出，是没有显著区别。
  4. "主模型变强 · 预处理塌缩" 范式 · 反复出现在 ML 历史里 · NLP 的 task head → prompt，TTS 的 phonemizer → end-to-end，VAD → micro-turn，OCR → VLM，手工 feature → 端到端 CNN/ViT。每次都是同种现象：外挂的预处理被主模型吸收。
  5. "端到端"的工程含义 · 整条 forward 过程中没有梯度断点 —— 任何中间表示都可以反向传播。dMel + flow matching 组合下 TML 实现真正的端到端：从音频输入到音频输出，loss 可以一路传回所有参数。codec-based 方案因为 codec 的离散步骤打断梯度（要靠 straight-through 近似），不算严格端到端。
  6. 2024-12 TML · Thinking Machines Lab 的 interaction model paper，用 dMel + flow matching 做实时语音 LLM。这是 dMel 在 frontier 模型里的第一次主要应用。证明"简单 + 端到端"组合在200ms 实时预算下能拿到生产可用质量。
  7. vocabulary 大小取舍 · dMel 默认 16 bin × 80 channel = 80 token/帧 · vocab 16 每个 token · 总信息量约等于 `80 × log2(16) = 320 bit/帧`。RVQ codec 通常 1-8 layer × 1024-2048 codebook · 总信息量约 100-300 bit/帧。同信息密度但 dMel 用更多更简单的 token 表达，RVQ 用更少更复杂的 token 表达。
  8. vmin / vmax 是数据集统计值 · dMel 不用学，但需要定 bin 范围。具体做法：跑过预训练数据集，取 log-mel 的 1% 和 99% 分位数作为 vmin / vmax（避开极端值）。等距分 K = 16 bin。这是个1 次性统计操作，跟"训练 codec"完全不同等级的复杂度。
  9. Vocoder · 把 mel 转回波形 · dMel 解决的是音频 → token方向。反向（token → 音频）仍需要 vocoder（HiFi-GAN / WaveNet / BigVGAN 等）。这部分跟 dMel 无关，可以是任何成熟方案。TML 的输出端用 flow matching 直接生成连续 mel，再过 vocoder。
  10. Pipeline 简化的工程价值 · 不只是代码量。简化带来：调试可视化（每 bin 对应已知能量范围）、版本控制简单（无 codec checkpoint）、新语言上线快（不用训新 codec）、A/B 实验便宜（改 bin 数立即试）。这些累计起来比 paper SOTA 重要。
  11. fish-speech 路线 vs dMel 路线 · 两种 speech-LLM 哲学。fish-speech 用 RVQ codec 让 LLM 见到"高质量音频抽象"，押注主模型用既有 next-token 能力学语音。dMel 让 LLM 见到"raw bin index 序列"，押注主模型够强能自己学复杂度。哲学差异 → 工程优先级差异。详见 audio-tokenization-rvq-vs-flow。
  12. "先做扎实简单方案"的工程哲学 · Apple / Anthropic 等公司有一个共同的工程文化：新点子先用最 naive 方案跑 ablation。如果简单方案不行，再加复杂度。这种"baseline-first" 思路在论文写作上不性感（"我们就是简化了一下"），但工程上极有价值 —— 帮人停下来想"我真的需要这个复杂度吗？"。dMel 就是这种文化的产物。
