给 LLM 加上记忆流 + 反思 + 规划，让 25 个 AI agent 在虚拟小镇里自主生活。会交朋友、传八卦、组织派对。现在几乎所有"有记忆的 AI agent"的祖宗。

TL;DR

LLM 本身没有记忆1 —— 聊完关窗口就什么都不记得。这篇论文给 LLM 外挂了三件套：记忆流2（所有观察按时间存下来）+ 反思3（定期从碎片里提炼高层认知）+ 规划4（每天早上生成计划，遇到新情况随时改）。25 个这样的 agent 放进一个像素风小镇5，涌现出了作者都没预料到的社会行为 —— 自发组织派对、信息在社交网络里扩散、关系随时间深化或疏远。

§ 01痛点 · LLM 没有记忆

你跟 ChatGPT 聊 3 个小时，关掉窗口，它什么都不记得。就算在一次对话里，context window 也有上限 —— 聊多了前面的就被挤掉。

这意味着 LLM 做不了"持续存在的角色"1：不记得昨天跟你聊了什么，不记得上周发生的事，不会因为积累了足够多的观察而"突然想通"某件事。

这篇论文的问题：怎么让 LLM 表现得像一个"活着的人" —— 有长期记忆、会总结经验、能根据过去的经历做决策、行为随时间变？

答案不是训一个更大的模型，是给现有 LLM 搭一套外挂系统 —— 记忆存在外面，需要的时候捞出来塞进 prompt。

§ 03记忆流 · 所有经历存成一条时间线

Agent 看到的、听到的、做的，全部按时间顺序存成一条流2。每条记忆带两个标签：时间戳和重要性评分（LLM 打分，1-10）。

记忆流跟数据库没什么区别 —— 就是一张只追加的表7。但它的威力在于**检索**的时候。

§ 05反思 · 碎片变认知

如果 agent 只存和检索原始观察，它永远只能做"碎片级别"的判断。反思3把一堆碎片**提炼成高层认知** —— 就像你从一个月的碎片观察里突然意识到"我组长要被换了"。

触发条件：不是定时触发，是最近一批记忆的重要性分数累积超过阈值10（比如加到 150 分就触发一次反思）。经历越密集越频繁 —— 像人一样，事多的时候想的也多。

§ 06Smallville · 涌现出来的社会行为

论文把 25 个这样的 agent 放进一个像素风小镇5，让它们自主生活两天。每个 agent 有名字、有职业、有初始人设描述（几句话）。剩下的全靠记忆 + 反思 + 规划自行演化。

§ 07工程视角 · 这套东西怎么落地

论文用的是 GPT-3.5，每步都调 API。25 个 agent 跑两天模拟花了几千美元 token 费12。直接照搬成本扛不住。

但概念框架每一块都能用更便宜的方式实现：

Memory Stream  →  PostgreSQL + pgvector（存观察 + embedding）
重要性评分     →  小模型打分 / 规则（不需要 GPT-4 级别）
检索          →  向量相似度 + 时间衰减 + 重要性加权（标准 RAG 管线）
反思          →  定时 batch job，用便宜模型做摘要
规划          →  每天调一次 LLM 生成日程，中间只在遇到大事时修改
行动          →  这步必须用好模型（决策质量直接影响用户体验）

核心思路：只在"行动"这一步用贵模型，其他步骤能省就省。记忆、检索、反思都可以用更轻的方案替代。

引用 / 术语表

  1. LLM 没有记忆 · LLM 每次推理都是无状态的 —— 它只看 prompt 里有的东西。上一轮对话的内容如果没塞进 prompt，就跟没发生过一样。context window 有长度限制（GPT-3.5 当时 4K token），更早的内容会被挤掉。
  2. Memory Stream · 记忆流 · 所有观察按时间顺序存成一条只追加的流，每条记忆带时间戳 + 重要性评分。是整个架构的数据基础 —— 检索、反思、规划都从这里取数据。详见 memory-stream。
  3. Reflection · 反思 · 定期把大量低层观察总结成高层认知。触发条件是重要性分数累积超过阈值。反思结果也存回记忆流，形成层级结构。详见 agent-reflection。
  4. Planning · 规划 · 每天早上让 LLM 基于 agent 的人设 + 最近记忆生成当天计划（"9 点画画，11 点去咖啡店，14 点找 Maria…"）。遇到计划外的事件（比如路上碰到人聊天）会修改计划。计划也存进记忆流。
  5. Smallville · 虚拟小镇 · 论文搭的实验环境 —— 一个像素风的 2D 小镇，有房屋、商店、公园、学校。25 个 agent 在里面自主生活。每个 agent 有名字、职业、初始人设描述（几句话），剩下全靠系统自行演化。
  6. 每步都调 LLM · 感知、检索排序、反思、规划、行动 —— 每一步都需要 LLM 参与。这是最大的成本瓶颈。25 个 agent × 每步 1 次 API call × 每天几百步 = 一天几千次 API call。
  7. 只追加的表 · Memory stream 不删除旧记忆（跟人脑不同 —— 人会遗忘）。靠检索时的时近度衰减来模拟"自然遗忘"效果 —— 老记忆不是被删了，是排序时权重低了。
  8. 检索排序 · 三维加权：时近度（指数衰减）× 重要性（LLM 打分）× 相关性（embedding 余弦相似度）。跟标准 RAG 的区别是多了时近度和重要性两个维度。详见 retrieval-scoring。
  9. RAG · Retrieval-Augmented Generation · 检索增强生成。核心思路是"先从外部知识库检索相关内容，再塞进 LLM prompt 辅助生成"。Generative Agents 的记忆检索本质上就是 RAG 的一种变体，但加了时间和重要性维度。
  10. 反思触发阈值 · 论文用重要性分数累积来决定何时触发反思。调大了 agent 反应迟钝（要积累很久才"想通"），调小了 token 费用高（太频繁地调 LLM 做摘要）。实际工程中这个阈值需要按场景调。
  11. 涌现行为 · 没有人编程让 agent 组织派对或传播信息。这些行为是从"记忆 + 反思 + 规划"的循环里自然长出来的。论文用人类评估验证了这些行为的"像人程度"显著高于没有记忆/反思的 baseline。
  12. Token 成本 · 25 个 agent 跑两天模拟消耗了几千美元的 GPT-3.5 API 费用。直接 scale 到更多 agent / 更长时间不现实。工程落地需要把检索、重要性评分、反思等步骤替换为更便宜的方案。
